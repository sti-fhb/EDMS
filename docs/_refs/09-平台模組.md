# 09 — 平台模組（DP）

**文件狀態：** 草稿（Draft）
**建立日期：** 2026-07-06
**來源：** 裁剪自 TBMS `_refs/08-資訊模組.md`，依 EDMS 2026-07-06 決策精簡（見文末「決策紀錄」）

> 本模組為 **EDMS 的共用平台層**，非業務模組。ET（教育訓練）與 DM（文件管理）共用其帳號、認證與稽核；各模組的**業務角色與權限各自維護**（權限分開做）。表名沿用 `DP_` 前綴。

---

## 1. 模組概述

平台模組（DP）提供 EDMS 全系統共用的基礎功能：**使用者帳號與登入認證、忘記密碼、密碼變更、使用者管理、帳號鎖定、系統參數、操作記錄（稽核）**。它不處理任何業務邏輯（課程、文件皆屬 ET / DM）。

### 1.1 定位與邊界

- **共用帳號、權限分開（SSO）**：ET 與 DM 共用同一份使用者帳號（`DP_USER`），使用者註冊 / 登入一次即可存取兩系統（SSO 指認證面：一次登入通用）；但**各模組的角色與權限完全由各模組自行維護**（ET 的管理者/教師/學員、DM 的各角色，分別放在 `ET_USER_ROLE` / `DM_USER_ROLE`）——**平台不碰角色**（不做角色指派、不做全域 RBAC）。
- **稽核集中（限資安類）**：登入登出、帳號建立/停用、角色/權限異動、系統參數異動等**資安稽核事件**，ET / DM 統一寫入本模組 `DP_AUDIT_LOG`（平台共用一張）；各模組之**業務歷程**（如 DM 文件變更歷程、閱讀紀錄、ET 學習/作答紀錄）仍留各模組、面向使用者查詢，不併入本表。
- **參數分層**：平台級參數（密碼策略、鎖定、帳號類信件模版等）放 `DP_PARAM`；模組自有參數（如 ET 的影片大小、排程時間）仍放各模組自己的參數表（如 `ET_PARAM`）。
- **排程基礎建設集中**：平台提供**單一排程執行引擎與 job 註冊機制**（多實例部署時含 leader 選舉，單一實例可簡化）；ET / DM 的排程作業（ET 週報/提醒、DM 寄信排程等）之 **job 邏輯放各模組**、向平台註冊執行——**平台不 import 業務模組**（依賴方向維持 模組 → 平台）。

### 1.2 操作角色

| 角色 | 說明 |
|------|------|
| 系統管理者 | 管理其他帳號（建立 / 停用 / 啟用 / 基本資料）、維護平台系統參數、查詢操作記錄（稽核）|
| 一般使用者 | 登入、登出、忘記密碼、變更密碼、維護自己的基本資料 |
| eMail Server | 外部郵件系統，寄發密碼重設信、帳號相關通知信 |

> ET / DM 的業務角色（管理者 / 教師 / 學員、DM 各角色）不在此定義；平台只認「帳號」與「是否為系統管理者」。

### 1.3 登入機制（簡單 JWT）

- 使用者於登入頁輸入帳號（Email）+ 密碼；驗證通過後系統核發一顆 **JWT access token**（TTL 由參數控制），前端後續請求以 `Authorization: Bearer <token>` 認證。
- **不採 MFA**（無二次驗證碼）、**不採 Refresh Token / Session Rotation**（無伺服器端 session 表）；登出＝前端丟棄 token。
- **無站點/工作站選擇、無 IP 設備識別**（EDMS 為單一組織，無多站點概念）。
- 帳號共用：同一帳號可登入 ET 與 DM；登入後由各模組依自己的角色決定可用功能與預設首頁。
- 新使用者**自助註冊即可用**（對齊 ET：註冊後自動取得該模組預設角色，如 ET 的「學員」）；管理者亦可代為建立帳號。

### 1.4 標準稽核欄位（全表適用）

EDMS 為單一組織，**不含站點/院區維度**。各表標準欄位統一為：

`CREATED_USER`、`CREATED_DATE`、`UPDATED_USER`、`UPDATED_DATE`、`RES_ID`、`DELETED`

- 預設**軟刪除**（`DELETED = 1`），查詢預設加 `WHERE DELETED = 0`
- append-only 記錄表（如 `DP_AUDIT_LOG`、`DP_PWD_HIST`）僅含 `CREATED_USER` / `CREATED_DATE`，無 `UPDATED_*` / `DELETED`

## 2. 功能需求

### 2.1 使用者登入

- 登入頁欄位：帳號（Email）、密碼。
- 驗證流程：
  1. 查 `DP_USER` 驗證帳密
     - 帳號不存在 → 提示「查無此帳號，請先註冊」
     - 密碼錯誤 → 累計失敗次數；達上限自動鎖定（見 §2.5）
     - 通過 → 核發 JWT，寫入登入操作記錄（`DP_AUDIT_LOG`，類別 LOGIN）
  2. 登入後由各模組依角色導向預設首頁（ET：管理者→ET07 / 教師→ET01 / 學員→ET04；DM 由 DM 決定）
- 登出：前端丟棄 token；寫登出記錄（類別 LOGOUT）。

### 2.2 忘記密碼

- 使用者於登入頁點「忘記密碼」，輸入帳號（Email）。
- 系統驗證帳號存在後，產生密碼重設 token（含有效期限，TTL 由參數控制，預設 30 分鐘），寄重設信至該 Email。
- **防帳號列舉**：不論帳號是否存在，皆回覆相同訊息（「若該 Email 已註冊，密碼重設信將寄至信箱」）。
- 使用者點信中連結 → 驗證 token 有效性與期限 → 輸入新密碼（驗證複雜度與重複性）→ 更新密碼、記錄操作歷程。

### 2.3 密碼變更

- 已登入使用者主動變更密碼：輸入舊密碼 + 新密碼 + 確認新密碼。
- 系統驗證舊密碼正確、新密碼兩次一致、符合複雜度與重複性規則後更新。
- 密碼一律以雜湊儲存（bcrypt 或等效），不可逆。

### 2.4 使用者 / 人員管理

- **自助註冊**：新使用者於登入頁註冊（帳號 Email、姓名、密碼），成功後自動取得所屬模組預設角色（如 ET 學員），即可登入。
- **管理者維護**：系統管理者可建立帳號、停用（Disable）/ 啟用（Enable）帳號、維護基本資料（姓名、Email）。
- **角色不在此**：ET / DM 的角色指派由各模組自己的權限管理頁處理（如 ET07）；平台使用者管理頁不設角色指派。
- 帳號（Email）變更採「新 Email 驗證後切換」延遲生效（對齊 ET US10：寄驗證信至新 Email，點連結後生效；期間舊 Email 仍可登入）。
- 帳號停用後於 ET / DM 兩端同步失效。
- 所有帳號異動寫入操作記錄。

> **已確認（2026-07-06）**：帳號啟用採「自助註冊即用」（對齊 ET US2）——註冊後即可登入、自動取得所屬模組預設角色；管理者代建之帳號亦**直接啟用可用**，不另寄「帳號開通確認信」。

### 2.5 帳號鎖定管理

- 連續登入失敗達設定次數（參數，預設 5）→ 自動鎖定帳號。
- 解鎖方式：**管理者手動解鎖** 或 **逾時自動解鎖**（等待時間為參數，預設 30 分鐘）。
- 鎖定 / 解鎖寫入操作記錄。

### 2.6 系統參數維護（平台級）

- 系統管理者維護平台共用參數（全域）。
- 操作：選參數分類 → 選參數項目 → 編輯值 → 系統驗證合法性後儲存 → 記錄異動歷程。
- 平台級參數見 §5；模組自有參數不在此（各模組自管）。

### 2.7 操作記錄（稽核）

- 系統記錄帳號建立、密碼重置、角色/資料異動、登入登出等操作歷程，供資安稽核。
- **平台共用一張** `DP_AUDIT_LOG`：ET / DM 的**資安稽核事件**（帳號 / 角色權限 / 系統操作）亦寫入本表統一查詢；業務歷程（DM 文件變更歷程等）不在此、留各模組。
- 記錄內容：序號（PK）、操作者、功能/模組代碼、操作類別（LOGIN / LOGOUT / CREATE / UPDATE / DELETE）、來源 IP、時間、異動對象代碼、異動前後值（JSONB）。
- 稽核日誌 **append-only**，不可竄改、不可刪除。
- 提供多條件查詢與匯出（CSV）。

### 2.8 排程執行基礎建設

- 平台提供**單一排程執行引擎**（如 APScheduler）與 **job 註冊介面**；多實例部署時以 leader 選舉確保只有一個實例觸發（單一實例可簡化）。
- 各模組於啟動時向平台**註冊自己的排程 job**（cron 表達式 + 執行函式）；**job 邏輯留在各模組**（ET 的 SCHET001/002、DM 的寄信排程），平台不 import 業務模組。
- 平台記錄**排程執行歷程**（起訖時間、成功/失敗、錯誤訊息）供查詢。
- 排程時間等參數由各模組自己的參數表控制（如 `ET_PARAM.WEEKLY_STAT_DAY_TIME`）；平台不做 DB 排程主檔與 UI 排程維護（採程式註冊）。

## 3. 資料模型（Table 清單）

> 完整欄位定義於後續 `docs/specs/dp/data-model.md`；此處為分析層清單。表名沿用 `DP_` 前綴，標準欄位見 §1.4。

| Table | Code | 檔案類別 | PK | 說明 |
|-------|------|---------|-----|------|
| 使用者主檔 | DP_USER | 主檔 | USER_ID | 帳號（Email）/ 密碼雜湊 / 姓名 / 狀態（啟用/停用）/ 登入失敗計數 / 鎖定狀態 / 最後登入；Email 變更之 PENDING 欄位（對齊 ET US10）|
| 密碼重設 Token | DP_PWD_RESET | 主檔 | TOKEN | 忘記密碼 / 帳號相關 token（TOKEN_TYPE 僅 `PWD_RESET`，如需帳號開通再加 `ACCOUNT_CONFIRM`）|
| 密碼歷程 | DP_PWD_HIST | 明細（append-only）| USER_ID + SEQ_NO | 前 N 次密碼雜湊，供密碼重複性檢核（精簡版）|
| 操作歷程 LOG | DP_AUDIT_LOG | 明細（append-only）| LOG_ID | 平台共用稽核日誌；ET / DM 皆寫入 |
| 功能參數主檔 | DP_PARAM_M | 主檔 | PARAM_ID | 平台級參數分類定義 |
| 功能參數明細 | DP_PARAM_D | 明細 | PARAM_ID + PARAM_KEY | 平台級參數實際值 |
| 排程執行歷程 | DP_SCHEDULE_LOG | 明細（append-only）| LOG_ID | 各模組排程 job 之執行紀錄（起訖/結果/錯誤）；**選配**，供查詢與補跑判斷 |

**明確不納入**（相對 TBMS 08 已裁剪）：

- ❌ `DP_SITE`（站點）/ 多站點架構 — EDMS 單一組織
- ❌ `DP_ROLE` / `DP_USER_ROLE` / `DP_MENU` / `DP_ROLE_MENUS`（全域 RBAC 與選單、角色指派）— 權限完全分開做，角色定義 / 指派 / 判定皆由各模組（ET/DM）自理，平台不碰
- ❌ `DP_SESSION`（伺服器端 session / Refresh Token）— 採無狀態簡單 JWT
- ❌ `DP_COMPDEVICE`（資訊設備 / IP 對照）、標籤印表機對照 — EDMS 不需要
- ❌ `DP_SCHEDULE`（DB 排程主檔）與 UI 排程維護 — 採**程式註冊**、排程時間放各模組參數表（**排程執行引擎與 job 註冊機制仍由平台提供**，見 §2.8；業務 job 邏輯歸各模組）
- ❌ MFA 相關（無二次驗證）
- ❌ 外部系統 API（APIDP001/002）— 目前無對外 API 需求

## 4. 跨模組介接

| 介接對象 | 介接方式 | 說明 |
|---------|---------|------|
| ET / DM | 共用帳號 | 兩模組共用 `DP_USER`；登入後各自依角色載入功能 |
| ET / DM | 資安稽核 | 各模組之資安稽核事件（帳號 / 角色權限 / 系統操作）統一寫入 `DP_AUDIT_LOG`；業務歷程留各模組 |
| ET / DM | 平台參數 | 密碼策略、鎖定、帳號類信件模版等由平台維護；模組自有參數各自管 |
| 外部 eMail 系統 | SMTP | 密碼重設信、帳號變更驗證信（ET / DM 的業務通知信仍由各模組自行寄送，走各自範本）|

> **對齊事項**：DM 的**資安類**日誌 `DM_USER_ROLE_LOG`（角色異動）改對齊平台 `DP_AUDIT_LOG`；DM 的**業務歷程** `DM_CHANGE_LOG`（文件變更）與閱讀紀錄 `DM_DOC_READ` **留 DM**（面向使用者、按文件查）。ET / DM data-model 的共用使用者表應統一為 `DP_USER`（型別/欄位以本模組為準）。

## 5. 可配置參數（平台級）

| 參數 | 說明 | 預設值建議 |
|------|------|-----------|
| JWT 有效時間 | access token TTL | （待定，如 8 小時）|
| 閒置自動登出（分鐘）| 閒置逾時自動登出（RQDP052：含敏感資料系統 ≤ 15 分）| 15 |
| 密碼最小長度 | 密碼複雜度策略 | 8 |
| 密碼字元組合要求 | 大小寫 / 數字 / 特殊符號 | 至少 3 種 |
| 密碼歷史記憶次數 | 禁止重複的前 N 次密碼 | 3 |
| 登入失敗鎖定次數 | 連續錯誤上限 | 5 |
| 帳號鎖定時間 | 自動解鎖等待（分鐘）| 30 |
| 密碼重設連結有效時間 | 忘記密碼 token 過期（分鐘）| 30 |
| eMail 模版-密碼重設 | 密碼重設信範本 | — |
| eMail 模版-帳號變更驗證 | 新 Email 驗證信範本 | — |

## 決策紀錄（2026-07-06）

| 項目 | 決定 | 說明 |
|------|------|------|
| 模組定位 | 平台共用層，非業務模組 | ET / DM 共用帳號/認證/稽核 |
| 名稱 / 前綴 | 沿用 `dp` / `DP_` | 與框架既有耦合（migrated code）一致，減少改動 |
| 認證 | 簡單 JWT，無 refresh、無 MFA | 同事確認 |
| 權限 | 完全分開（各模組自理角色定義 / 指派 / 判定）| 平台只提供帳號，不碰角色、不做全域 RBAC / 選單 |
| 稽核 | 平台共用一張 `DP_AUDIT_LOG` | ET / DM 皆寫入 |
| 標準欄位 | 去 SITE / HOSPITAL | EDMS 單一組織；一併解決 ET/DM 標準欄位不一致 |
| 帳號啟用 | 自助註冊即用（對齊 ET）| 管理者代建之帳號亦直接可用，不寄開通確認信 |
| 密碼歷程表 | 納入 `DP_PWD_HIST` 精簡版 | 供密碼重複性檢核（禁止與前 N 次相同）|
| 排程 | 執行基礎建設集中於平台，業務 job 歸各模組（方案 B）| 平台提供排程引擎 + job 註冊 + 執行 log；ET/DM 的 job 邏輯各自放並向平台註冊，平台不反向 import 業務模組 |
