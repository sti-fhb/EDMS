# 09 — 平台模組（DP）

**文件狀態：** 草稿（Draft）
**建立日期：** 2026-07-06
**來源：** 裁剪自 TBMS `_refs/08-資訊模組.md`，依 EDMS 2026-07-06 決策精簡（見文末「決策紀錄」）

> 本模組為 **EDMS 的共用平台層**，非業務模組。ET（教育訓練）與 DM（文件管理）共用其帳號、認證與稽核；各模組的**業務角色與權限各自維護**（權限分開做）。表名沿用 `DP_` 前綴。

---

## 1. 模組概述

平台模組（DP）提供 EDMS 全系統共用的基礎功能：**使用者帳號與登入認證、忘記密碼、密碼變更、使用者管理、帳號鎖定、系統參數（全平台集中，前綴分模組）、通知範本與發信引擎、排程引擎、操作記錄（稽核）**。它不處理任何業務邏輯（課程、文件皆屬 ET / DM）——它管的是「內容無關的基礎設施」與「共用帳號操作」，各模組的業務內容（參數值、範本文案、排程 job 邏輯）仍由各模組作者維護。

### 1.1 定位與邊界

- **共用帳號、權限分開（SSO）**：ET 與 DM 共用同一份使用者帳號（`DP_USER`），使用者註冊 / 登入一次即可存取兩系統（SSO 指認證面：一次登入通用）；但**各模組的角色與權限完全由各模組自行維護**（ET 的管理者/教師/學員、DM 的各角色，分別放在 `ET_USER_ROLE` / `DM_USER_ROLE`）——**平台不碰角色**（不做角色指派、不做全域 RBAC）。
- **稽核集中（限資安類）**：登入登出、帳號建立/停用、角色/權限異動、系統參數異動等**資安稽核事件**，ET / DM 統一寫入本模組 `DP_AUDIT_LOG`（平台共用一張）；各模組之**業務歷程**（如 DM 文件變更歷程、閱讀紀錄、ET 學習/作答紀錄）仍留各模組、面向使用者查詢，不併入本表。
- **系統參數集中、前綴分模組（2026-07-08 改）**：全平台系統參數統一存平台 `DP_PARAM_M/D`，以 `PARAM_ID` 前綴（`DP_` / `ET_` / `DM_`）區分歸屬；平台提供唯讀查詢服務供各模組讀取。**維護 UI 仍留各模組**（各自過濾自己前綴，管理者只看得到自己模組的參數），DP 系統參數頁只顯示平台級 `DP_` 參數。（原「參數分層、模組各自持 `ET_PARAM` / `DM_PARAM`」已廢除——理由見決策紀錄）
- **通知範本與發信引擎集中（2026-07-08 新增）**：通知範本統一存平台 `DP_NOTIFY_TEMPLATE`（`MODULE` 欄 `DP` / `ET` / `DM` 區分歸屬），平台提供**唯一發信服務**與寄件 outbox（`DP_EMAIL_LOG`）；不同模組、內容不同的通知（課程邀請、文件送審、各自週報…）＝同表內不同 `TEMPLATE_CODE` 列。**範本編輯 UI 留各模組**（過濾自己 MODULE）。DP 自身固定系統信（密碼重設 / 帳號變更驗證）列 `MODULE=DP`、不可刪。（原「各模組自持 `ET_NOTIFY_TEMPLATE` / `DM_NOTIFY_TEMPLATE` / `DM_NOTIFY_QUEUE`」已廢除）
- **排程集中：引擎 + DB 註冊表（2026-07-08 升級）**：平台提供單一排程引擎（APScheduler，多實例含 leader 選舉、單一實例可簡化）＋ `DP_SCHEDULE` 註冊表（cron + 執行程式參照 + 啟停）＋ `DP_SCHEDULE_LOG` 執行歷程；各模組 job handler 向引擎註冊（ET 週報/提醒 SCHET001/002、DM KPI 週報 SCHDM001），排程時間等參數存 `DP_PARAM`（前綴分模組）。（原「純程式註冊、不建 DB 主檔、排程時間放各模組參數表」升級為 DB 註冊表集中，對齊 TBMS 核心模組）

### 1.2 操作角色

| 角色 | 說明 |
|------|------|
| 平台管理員 | DP 自身功能之把關者（IT）：管理其他帳號（建立 / 停用 / 啟用 / 基本資料）、維護平台級 `DP_` 參數、查詢全平台操作記錄（稽核）。以 `DP_USER.IS_PLATFORM_ADMIN` 旗標判定，**非業務角色體系**（不與 ET / DM 角色混用）|
| 一般使用者 | 登入、登出、忘記密碼、變更密碼、維護自己的基本資料 |
| eMail Server | 外部郵件系統，平台發信引擎透過它寄發所有信件（密碼重設 / 帳號驗證 / 各模組業務通知）|

> ET / DM 的業務角色（管理者 / 教師 / 學員、DM 各角色）不在此定義；平台只認「帳號」與「是否為平台管理員」。
> **平台管理員 vs 模組管理者**：平台管理員（IT）只管 DP 自身（帳號 / 平台參數 / 稽核），**看不到 ET / DM 的業務參數與範本**（那些在各模組系統設定頁、由各模組管理者維護）；反之 ET 管理者、DM 管理者也各管各的、互不可見。集中的只是「儲存與引擎」，可見範圍仍按模組隔離。

### 1.3 登入機制（簡單 JWT）

- 使用者於登入頁輸入帳號（Email）+ 密碼；驗證通過後系統核發一顆 **JWT access token**（TTL 由參數控制），前端後續請求以 `Authorization: Bearer <token>` 認證。
- **不採 MFA**（無二次驗證碼）、**不採 Refresh Token / Session Rotation**（無伺服器端 session 表）；登出＝前端丟棄 token。
- **無站點/工作站選擇、無 IP 設備識別**（EDMS 為單一組織，無多站點概念）。
- 帳號共用：同一帳號可登入 ET 與 DM；登入後由各模組依自己的角色決定可用功能與預設首頁。
- 新使用者**自助註冊即可用**（對齊 ET：註冊後自動取得該模組預設角色，如 ET 的「學員」）；管理者亦可代為建立帳號。

### 1.4 標準稽核欄位（全表適用）

EDMS 為單一組織，**不含站點/院區維度**。各表標準欄位統一為：

`CREATED_USER`、`CREATED_DATE`、`UPDATED_USER`、`UPDATED_DATE`、`RES_ID`、`DELETED`

- 預設**軟刪除**（`DELETED = 1`），查詢預設加 `WHERE DELETED = 0`
- append-only 記錄表（如 `DP_AUDIT_LOG`、`DP_PWD_HIST`）僅含 `CREATED_USER` / `CREATED_DATE`，無 `UPDATED_*` / `DELETED`

## 2. 功能需求

### 2.1 使用者登入

- 登入頁欄位：帳號（Email）、密碼。
- 驗證流程：
  1. 查 `DP_USER` 驗證帳密
     - 帳號不存在 → 提示「查無此帳號，請先註冊」
     - 密碼錯誤 → 累計失敗次數；達上限自動鎖定（見 §2.5）
     - 通過 → 核發 JWT，寫入登入操作記錄（`DP_AUDIT_LOG`，類別 LOGIN）
  2. 登入後由各模組依角色導向預設首頁（ET：管理者→ET07 / 教師→ET01 / 學員→ET04；DM 由 DM 決定）
- 登出：前端丟棄 token；寫登出記錄（類別 LOGOUT）。

### 2.2 忘記密碼

- 使用者於登入頁點「忘記密碼」，輸入帳號（Email）。
- 系統驗證帳號存在後，產生密碼重設 token（含有效期限，TTL 由參數控制，預設 30 分鐘），寄重設信至該 Email。
- **防帳號列舉**：不論帳號是否存在，皆回覆相同訊息（「若該 Email 已註冊，密碼重設信將寄至信箱」）。
- 使用者點信中連結 → 驗證 token 有效性與期限 → 輸入新密碼（驗證複雜度與重複性）→ 更新密碼、記錄操作歷程。

### 2.3 密碼變更

- 已登入使用者主動變更密碼：輸入舊密碼 + 新密碼 + 確認新密碼。
- 系統驗證舊密碼正確、新密碼兩次一致、符合複雜度與重複性規則後更新。
- 密碼一律以雜湊儲存（bcrypt 或等效），不可逆。

### 2.4 使用者 / 人員管理

- **自助註冊**：新使用者於登入頁註冊（帳號 Email、姓名、密碼），成功後自動取得所屬模組預設角色（如 ET 學員），即可登入。
- **管理者維護**：平台管理員可建立帳號、停用（Disable）/ 啟用（Enable）帳號、維護基本資料（姓名、Email）。
- **角色不在此**：ET / DM 的角色指派由各模組自己的權限管理頁處理（如 ET07）；平台使用者管理頁不設角色指派。
- 帳號（Email）變更採「新 Email 驗證後切換」延遲生效（對齊 ET US10：寄驗證信至新 Email，點連結後生效；期間舊 Email 仍可登入）。
- 帳號停用後於 ET / DM 兩端同步失效。
- 所有帳號異動寫入操作記錄。

> **已確認（2026-07-06）**：帳號啟用採「自助註冊即用」（對齊 ET US2）——註冊後即可登入、自動取得所屬模組預設角色；管理者代建之帳號亦**直接啟用可用**，不另寄「帳號開通確認信」。

### 2.5 帳號鎖定管理

- 連續登入失敗達設定次數（參數，預設 5）→ 自動鎖定帳號。
- 解鎖方式：**管理者手動解鎖** 或 **逾時自動解鎖**（等待時間為參數，預設 30 分鐘）。
- 鎖定 / 解鎖寫入操作記錄。

### 2.6 系統參數（全平台集中，前綴分模組）

- **儲存集中**：全平台系統參數（平台級 + ET / DM 模組級）統一存 `DP_PARAM_M/D`，以 `PARAM_ID` 前綴（`DP_` / `ET_` / `DM_`）區分歸屬。
- **唯讀服務**：平台提供參數查詢服務（`get_param_values(param_id, key)`），供各模組讀取自己的參數值；寫入僅由各自維護 UI。
- **維護 UI 按模組隔離**：
  - 平台管理員於 DP 系統參數頁維護、且**只看得到平台級 `DP_` 參數**（JWT/閒置逾時、密碼政策、鎖定門檻、發信引擎參數等）。
  - ET / DM 的模組參數（如 `ET_VIDEO_MAX_SIZE_MB`、`DM_REMIND_THRESHOLD`）由各模組管理者於自己的系統設定頁維護，DP 頁**不顯示**。
- 操作：選參數項目 → 編輯值 → 系統驗證合法性後儲存 → 記錄異動歷程（稽核）。
- 平台級參數清單見 §5.1；模組級參數（存於 DP_PARAM、由各模組維護）見 §5.2。

### 2.7 通知範本與發信引擎（集中）

- **範本集中**：所有通知範本存 `DP_NOTIFY_TEMPLATE`，`MODULE` 欄（`DP` / `ET` / `DM`）+ `TEMPLATE_CODE` 區分；欄位含主旨、內文、可用變數、管道（Email / 站內 / 兩者）、啟用停用、版本（樂觀鎖）。內容不同的通知＝同表不同列。
- **發信引擎集中**：平台提供唯一發信服務 `send_email(recipients, template_code, params)` 與寄件 outbox `DP_EMAIL_LOG`（非同步批次、狀態、重試、`CALLER_MODULE`）；各模組只呼叫發信服務並傳 `template_code`，不自持範本表、不自建 outbox。
- **編輯 UI 按模組隔離**：ET 管理者於 ET 系統設定編輯 `MODULE=ET` 的範本、DM 管理者編輯 `MODULE=DM`；DP 自身系統信（密碼重設 / 帳號變更驗證，`MODULE=DP`）由**平台管理員**於 DP 系統參數頁維護（可編輯主旨 / 內文），但**不可停用 / 刪除**（帳號安全信），各模組亦不可編輯。
- 發信引擎之調校參數（重試上限、每分鐘速率、失敗告警比例）屬平台級 `DP_` 參數（見 §5.1）。

### 2.8 操作記錄（稽核）

- 系統記錄帳號建立、密碼重置、角色/資料異動、登入登出等操作歷程，供資安稽核。
- **平台共用一張** `DP_AUDIT_LOG`：ET / DM 的**資安稽核事件**（帳號 / 角色權限 / 系統操作）亦寫入本表統一查詢；業務歷程（DM 文件變更歷程等）不在此、留各模組。
- 記錄內容：序號（PK）、操作者、功能/模組代碼、操作類別（LOGIN / LOGOUT / CREATE / UPDATE / DELETE）、來源 IP、時間、異動對象代碼、異動前後值（JSONB）。
- 稽核日誌 **append-only**，不可竄改、不可刪除。
- 提供多條件查詢與匯出（CSV）。

### 2.9 排程執行基礎建設（引擎 + DB 註冊表，集中）

- 平台提供**單一排程執行引擎**（如 APScheduler）＋ **`DP_SCHEDULE` 註冊表**（`JOB_ID`、cron 表達式、執行程式參照、啟停、上次執行時間 / 結果）；多實例部署時以 leader 選舉確保只有一個實例觸發（單一實例可簡化）。
- 排程作業：ET 週報 / 未看提醒（SCHET001）、ET 到期關閉 + 加急提醒（SCHET002）、DM KPI 週報 + 未讀提醒（SCHDM001）等，於註冊表登錄 job 定義；job handler 由各模組提供並向引擎註冊。
- 平台記錄 **`DP_SCHEDULE_LOG` 排程執行歷程**（起訖時間、成功/失敗、錯誤訊息）供查詢與補跑判斷。
- 排程時間等參數存 `DP_PARAM`（前綴分模組，如 `ET_WEEKLY_STAT_DAY_TIME`、`DM_WEEKLY_SCHED_DAY_TIME`），由各模組於自己的系統設定頁維護。
- **依賴方向**：job 需要業務資料時反向 import 各模組 service（平台 job → 模組 service）；實作層 job 程式碼落點於 `docs/specs/dp/` 定案（可比照 TBMS 集中於平台命名空間 + 白名單）。

## 3. 資料模型（Table 清單）

> 完整欄位定義於後續 `docs/specs/dp/data-model.md`；此處為分析層清單。表名沿用 `DP_` 前綴，標準欄位見 §1.4。

| Table | Code | 檔案類別 | PK | 說明 |
|-------|------|---------|-----|------|
| 使用者主檔 | DP_USER | 主檔 | USER_ID | 帳號（Email）/ 密碼雜湊 / 姓名 / 狀態（啟用/停用）/ 登入失敗計數 / 鎖定狀態 / 最後登入 / **平台管理員旗標 `IS_PLATFORM_ADMIN`**；Email 變更之 PENDING 欄位（對齊 ET US10）|
| 密碼重設 Token | DP_PWD_RESET | 主檔 | TOKEN | 忘記密碼 / 帳號相關 token（TOKEN_TYPE 僅 `PWD_RESET`，如需帳號開通再加 `ACCOUNT_CONFIRM`）|
| 密碼歷程 | DP_PWD_HIST | 明細（append-only）| USER_ID + SEQ_NO | 前 N 次密碼雜湊，供密碼重複性檢核（精簡版）|
| 操作歷程 LOG | DP_AUDIT_LOG | 明細（append-only）| LOG_ID | 平台共用稽核日誌；ET / DM 皆寫入 |
| 功能參數主檔 | DP_PARAM_M | 主檔 | PARAM_ID | 全平台參數分類定義（前綴 `DP_`/`ET_`/`DM_` 分模組）|
| 功能參數明細 | DP_PARAM_D | 明細 | PARAM_ID + PARAM_KEY | 全平台參數實際值 |
| 通知範本 | DP_NOTIFY_TEMPLATE | 主檔 | MODULE + TEMPLATE_CODE | 全平台通知範本（`MODULE` 分 `DP`/`ET`/`DM`）；主旨 / 內文 / 變數 / 管道 / 啟停 / 版本 |
| 寄件 outbox / 發信 LOG | DP_EMAIL_LOG | 明細（append-only）| MESSAGE_ID | 平台發信引擎之寄件佇列與結果（狀態 / 重試 / 錯誤 / `CALLER_MODULE`）；取代 DM_NOTIFY_QUEUE |
| 排程註冊表 | DP_SCHEDULE | 主檔 | JOB_ID | 排程定義（cron / 執行程式 / 啟停 / 上次執行）；各模組 job 於此登錄 |
| 排程執行歷程 | DP_SCHEDULE_LOG | 明細（append-only）| LOG_ID | 各模組排程 job 之執行紀錄（起訖/結果/錯誤）；供查詢與補跑判斷 |

**明確不納入**（相對 TBMS 08 已裁剪）：

- ❌ `DP_SITE`（站點）/ 多站點架構 — EDMS 單一組織
- ❌ `DP_ROLE` / `DP_USER_ROLE` / `DP_MENU` / `DP_ROLE_MENUS`（全域 RBAC 與選單、業務角色指派）— 權限完全分開做，角色定義 / 指派 / 判定皆由各模組（ET/DM）自理，平台不碰（平台僅有 `IS_PLATFORM_ADMIN` 旗標把關 DP 自身功能，非業務角色體系）
- ❌ `DP_SESSION`（伺服器端 session / Refresh Token）— 採無狀態簡單 JWT
- ❌ `DP_COMPDEVICE`（資訊設備 / IP 對照）、標籤印表機對照 — EDMS 不需要
- ❌ MFA 相關（無二次驗證）
- ❌ 外部系統 API（APIDP001/002）— 目前無對外 API 需求

> **2026-07-08 納入調整**：`DP_NOTIFY_TEMPLATE`、`DP_EMAIL_LOG`、`DP_SCHEDULE`（DB 註冊表）由「原不納入 / 各模組自持」改為**納入平台集中**（參數 / 範本 / 排程集中決策，見決策紀錄）。連帶廢除各模組 `ET_PARAM` / `DM_PARAM` / `ET_NOTIFY_TEMPLATE` / `DM_NOTIFY_TEMPLATE` / `DM_NOTIFY_QUEUE`。

## 4. 跨模組介接

| 介接對象 | 介接方式 | 說明 |
|---------|---------|------|
| ET / DM | 共用帳號 | 兩模組共用 `DP_USER`；登入後各自依角色載入功能 |
| ET / DM | 資安稽核 | 各模組之資安稽核事件（帳號 / 角色權限 / 系統操作）統一寫入 `DP_AUDIT_LOG`；業務歷程留各模組 |
| ET / DM | 系統參數 | 全平台參數存 `DP_PARAM`（前綴分模組）；平台提供唯讀查詢服務，各模組讀自己前綴；維護 UI 各模組自理 |
| ET / DM | 通知範本 + 發信 | 範本存 `DP_NOTIFY_TEMPLATE`（`MODULE` 分模組）；各模組呼叫平台唯一發信服務並傳 `template_code`，經 `DP_EMAIL_LOG` outbox 非同步寄送 |
| ET / DM | 排程 | 各模組 job 於 `DP_SCHEDULE` 註冊、由平台引擎執行，`DP_SCHEDULE_LOG` 記錄結果；job 需要業務資料時反向 import 模組 service |
| 外部 eMail 系統 | SMTP | 平台發信引擎透過它寄發**所有**信件（密碼重設 / 帳號驗證 / ET / DM 業務通知），統一走 `DP_NOTIFY_TEMPLATE` 範本 |

> **對齊事項**：
> - DM 的**資安類**日誌 `DM_USER_ROLE_LOG`（角色異動）改對齊平台 `DP_AUDIT_LOG`；DM 的**業務歷程** `DM_CHANGE_LOG`（文件變更）與閱讀紀錄 `DM_DOC_READ` **留 DM**（面向使用者、按文件查）。ET / DM data-model 的共用使用者表統一為 `DP_USER`（型別/欄位以本模組為準）。
> - **參數 / 範本 / 排程集中（2026-07-08）**：ET / DM data-model 應**移除** `ET_PARAM` / `DM_PARAM`（改存 `DP_PARAM`，前綴 `ET_` / `DM_`）、`ET_NOTIFY_TEMPLATE` / `DM_NOTIFY_TEMPLATE`（改存 `DP_NOTIFY_TEMPLATE`，`MODULE=ET`/`DM`）、`DM_NOTIFY_QUEUE`（改用 `DP_EMAIL_LOG`）；排程改掛 `DP_SCHEDULE`。密碼重設 TTL 等認證相關參數改列平台級 `DP_`。

## 決策紀錄（2026-07-08 — 參數 / 範本 / 排程集中化）

> **背景**：查證 TBMS 分工後確認——核心業務模組（BS/BC/CP/MA/TL）之參數 / 範本 / 排程本就集中於 DP，僅後期獨立子系統 ET / DM 為避免耦合外部大平台而自持。EDMS 的 DP 是**自家平台**、ET / DM 已共用 DP_USER / DP_AUDIT_LOG / DP 認證，ET / DM 當初自持的理由不成立，故**改為集中**、對齊 TBMS 核心模組。

| 項目 | 決定 | 說明 |
|------|------|------|
| 系統參數 | **集中 `DP_PARAM_M/D`，前綴分模組** | 廢除 `ET_PARAM` / `DM_PARAM`；`PARAM_ID` 前綴 `DP_`/`ET_`/`DM_` 區分；平台提供唯讀查詢服務。密碼重設 TTL、發信引擎調校等改列平台級 `DP_` |
| 通知範本 | **集中 `DP_NOTIFY_TEMPLATE`，`MODULE` 分模組** | 廢除 `ET_NOTIFY_TEMPLATE` / `DM_NOTIFY_TEMPLATE`；schema 相同、內容差異＝同表不同 `TEMPLATE_CODE` 列 |
| 發信引擎 | **集中：平台唯一發信服務 + `DP_EMAIL_LOG` outbox** | 廢除 `DM_NOTIFY_QUEUE`；各模組只傳 `template_code`；連 DP 自身系統信（`MODULE=DP`）一併走此引擎 |
| 排程 | **升級：`DP_SCHEDULE` DB 註冊表 + 引擎 + `DP_SCHEDULE_LOG`** | 由方案 B（純程式註冊、不建 DB 主檔）升級為 DB 註冊表集中；job 需要業務資料時反向 import 模組 service |
| 維護 UI | **留各模組、按模組隔離** | ET 管理者只看 ET 參數/範本、DM 管理者只看 DM，互不可見；DP 系統參數頁只顯示平台級 `DP_`。集中的只是儲存與引擎 |
| 平台管理員 | **`DP_USER.IS_PLATFORM_ADMIN` 旗標（最小）** | 只把關 DP 自身功能（帳號 / 平台參數 / 稽核）；非業務角色體系，不與 ET / DM 角色混用；**看不到** ET / DM 業務參數與範本 |
| 業務設定不動 | ET 標籤庫、DM 分類 / func_name / 可見對象、各模組角色指派 | 皆為業務語意，**完全留各模組**，不受本次集中影響 |
