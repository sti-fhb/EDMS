## 模組定位

平台模組（DP）為 EDMS 全系統之**共用基礎層 + 統一管理後台**，非業務模組。ET（教育訓練）與 DM（文件管理）共用其**帳號、認證、稽核**，以及系統參數 / 清單定義 / 通知範本 / 排程之**儲存與引擎**；各模組的業務角色**能力與判定**、以及關聯 / 指派資料（誰配哪個標籤 / 角色 / 分類）仍留各模組（權限分開做）。表名沿用 `DP_` 前綴。

涵蓋能力：使用者登入（簡單 JWT）/ 登出、自助註冊、忘記密碼、個人資料維護（含密碼變更、Email 變更驗證切換）、使用者 / 帳號管理、帳號鎖定、**系統參數與清單定義（全平台集中、前綴分模組）、通知範本與發信引擎、排程引擎、權限管理（角色 + 標籤 / 可見對象指派）**、操作記錄（稽核）。

> **集中化決策（2026-07-08）**：所有管理畫面（系統參數與清單、通知範本、權限管理、帳號管理、操作記錄）集中於 DP 統一管理後台、按模組過濾；**「有哪些」的定義（分類 / 標籤名稱 / func_name / 參數值）存 `DP_PARAM`、「誰配誰」的關聯 / 指派與角色判定留各模組**。平台不做全域 RBAC（角色能力與判定由各模組 enforce）。ET / DM 之登入 / 註冊 / 忘記密碼 / 個資畫面均由 DP 提供，各模組不自設（權威見 [`_refs/09-平台模組.md`](../../_refs/09-平台模組.md)）。

### 主要角色

平台**不另設**「平台管理員」；DP 後台由 ET / DM 管理者進入，靠模組過濾決定可見與可改內容。

| 角色 | 職責 |
|------|------|
| ET 管理者 | ET 業務角色（`ET_USER_ROLE`）。於 DP 後台維護 ET 相關（前綴 `ET_` / `MODULE=ET`）之參數 / 清單 / 範本 / ET 角色與標籤指派；並可存取共用項 |
| DM 管理者 | DM 業務角色（`DM_USER_ROLE`）。於 DP 後台維護 DM 相關（前綴 `DM_` / `MODULE=DM`）之參數 / 清單 / 範本 / DM 角色與可見對象指派；並可存取共用項 |
| 一般使用者 | 登入 / 登出、自助註冊、忘記密碼、維護自己的基本資料（姓名 / Email / 密碼）|
| eMail Server | 外部郵件系統；平台發信引擎透過它寄發所有信件（密碼重設 / 帳號驗證 / 各模組業務通知）|

> **可見與存取規則**：**模組項**（前綴 `ET_` / `DM_` 之參數與清單、`MODULE=ET` / `DM` 之範本、各模組角色與標籤指派）按模組隔離、**互不可見**；**共用項**（帳號管理、平台級無前綴 `DP_` 參數、操作記錄稽核）不分模組，**ET / DM 管理者皆可存取**。
>
> **取捨**：帳號與平台級參數為共用，故 ET 管理者可停用帳號（影響 DM）、可改密碼策略（影響全平台）、可查全部稽核——為換取「單一後台、無獨立 IT 角色」之簡化，接受此共管範圍（2026-07-08 決策）。

## Clarifications

### Session 2026-07-09（spec.md 釐清第 4 輪——首次註冊使用者之引導）

- Q: 新註冊使用者（僅 ET 學員）於入口頁如何看待 DM？ → A: **DM 卡恆顯示**——無任何 DM 角色者呈「**未開通**」鎖定狀態（灰卡 + 鎖頭 + 引導文字「請洽您的單位主管或管理者開通」，點擊僅提示不進入）；只揭露模組存在、不揭露任何文件內容（存取控制仍由 DM 端 enforce）。**取代第 3 輪「DM 入口僅具角色者顯示」**
- Q: 是否做「開通申請流程」（註冊後選模組 → 系統通知管理者）？ → A: **v1 不做**。以歡迎橫幅 + 鎖定卡引導使用者線下洽單位主管 / 管理者（內部系統、開通成本低）；「申請開通」按鈕（寄信通知該模組全部管理者、不建申請單、不追蹤狀態）列為**未來擴充 backlog**
- Q: 首次登入是否需要引導？ → A: 入口頁顯示**歡迎橫幅一次**（「您已可使用教育訓練（學員）；如需文件管理或其他權限請洽管理者」，可關閉；「已顯示過」旗標之儲存位置於實作定）
- Q: ET / DM 介面內側欄之跨模組可見性？ → A: 教育訓練功能組**人人可見**；文件管理功能組**僅具任一 DM 角色者可見**（未開通＝隱藏、非鎖定顯示——引導職責歸入口頁鎖定卡，側欄保持乾淨）；系統管理後台組僅 ET / DM 管理者可見；由各模組介面 enforce
- Q: 「自 ET / DM 被導來登入」的語意？ → A: 精確化——**因未登入 / 閒置逾時而被攔到登入頁者**（通知信連結、書籤、逾時重登）登入後**返回原目標頁面**；無目標頁面者（含新註冊使用者）才進入口頁。逾時重登（15 分鐘閒置）為最頻繁情境，redirect 返回原頁面為必要 UX

### Session 2026-07-08（spec.md 釐清第 3 輪）

- Q: 新使用者的預設權限？ → A: 帳號建立當下（自助註冊 / 管理者代建）即自動授予 **ET 學員**（唯一預設角色，透過 ET service 寫入 `ET_USER_ROLE`）；**其他角色（ET 教師 / 管理者、DM 全部四角色〔含閱覽者〕）一律由管理者於權限管理（US7）開通**，未開通者看不到相關功能。**本項推翻第 1 輪「註冊不授角色、首次進入各模組時各自授予預設角色」之決議**；DM「首次登入自動授予閱覽者」規則同步作廢（DM 相關文件需回頭同步）
- Q: 模組入口頁如何配合？ → A: ET 入口恆顯示（人人具 ET 學員）；個人資料維護恆顯示〔DM 卡顯示方式**已被第 4 輪改為「未開通鎖定卡」**，以第 4 輪為準〕

### Session 2026-07-08（spec.md 釐清第 2 輪）

- Q: 發信失敗 / 稽核容量之「通報管理者」如何實作？ → A: **不做系統內通報功能**；發信失敗率、稽核容量等系統性告警由 **IT 監控機制**（log / 監控工具）負責；系統僅如實記錄狀態（FAILED 標記、容量自動因應動作留軌跡）
- Q: 密碼效期（90 日）到期後之行為？ → A: **到期後登入時強制變更**密碼方可繼續使用；到期前由平台排程 `SCHDP001` 寄 Email 提醒（預設到期前 7 天起，天數為平台級參數；範本 `MODULE=DP`「密碼到期提醒」）
- Q: 防範自動化程式登入嘗試之具體機制？ → A: 對登入 / 忘記密碼 / 密碼變更端點實施**伺服器端速率限制**（以來源 IP + 帳號為維度）；**不採 CAPTCHA**（內部系統、避免影響易用性）；帳號鎖定機制照舊
- Q: 管理者如何進入 DP 後台？ → A: **僅從 ET / DM 介面內之管理選單連結進入**；模組入口頁維持三項（ET / DM 入口 + 個人資料維護），不顯示後台入口

### Session 2026-07-08（spec.md 釐清第 1 輪）

- Q: 管理者代建帳號的初始密碼如何給定？「預設密碼」「特權帳號」適用誰？ → A: 代建時由管理者設定一組初始密碼並標記「**首次登入強制變更**」；系統內「預設密碼」僅此情境（自助註冊由使用者自訂密碼）。「**特權帳號**」＝具 ET / DM 管理者角色之帳號；12 字元門檻於其**下次變更密碼時**適用（被指派管理者角色當下不強制改密碼）
- Q: 註冊後「所屬模組預設角色」何時授予？登入後看到什麼？ → A: 〔角色授予部分**已被第 3 輪推翻**、導向與入口頁呈現**已被第 4 輪精確化**（redirect 語意、DM 鎖定卡），以後者為準〕原決議：登入後有來源返回原模組；無來源顯示模組入口頁（ET / DM 入口 + 個人資料維護）
- Q: 閒置自動登出如何實作？JWT TTL 8 小時與「敏感系統 ≤ 15 分」之關係？ → A: EDMS 認定為**含敏感資料系統**；採**短 TTL + 前端活動換發**——token TTL 即閒置逾時（預設 15 分鐘），前端有操作即靜默換發新 token、停止操作逾時自然失效；「8 小時」改定義為**單日換發上限**（自本次登入起算，換發不得超過，逾上限需重新登入）
- Q: 範本管道「站內」由誰實作？ → A: 站內訊息之儲存表與 UI **留各模組自理**；DP 範本之管道欄位僅作為該事件**是否寄 Email** 之開關依據，站內文案是否引用範本由模組自行決定；DP 不設站內訊息表
- Q: 排程總覽畫面與平台自身排程作業？ → A: US11 納入「**排程總覽**」唯讀畫面（共用項；列 job / cron / 啟停 / 上次執行結果；不可於 UI 啟停或手動補跑，啟停由 DB / 部署管理）；閒置帳號禁用（90 日）與密碼效期提醒登錄為平台自身排程 **`SCHDP001`（每日）**；outbox 背景 worker 為常駐程序、不入排程表；稽核每日備份屬 IT 維運、不在系統功能範圍

### Session 2026-07-08（集中化與統一管理後台決策）

- Q: 系統參數 / 清單 / 通知範本 / 排程由各模組自持或集中平台？ → A: **集中**。參數與清單定義統一存 `DP_PARAM_M/D`（`PARAM_ID` 前綴 `DP_` / `ET_` / `DM_` 分歸屬）、範本統一存 `DP_NOTIFY_TEMPLATE`（`MODULE` 分模組）、寄送統一走平台發信服務 + outbox `DP_EMAIL_LOG`、排程統一 `DP_SCHEDULE` 註冊表 + 平台引擎 + `DP_SCHEDULE_LOG`；連帶廢除各模組 `ET_PARAM` / `DM_PARAM` / `ET_NOTIFY_TEMPLATE` / `DM_NOTIFY_TEMPLATE` / `DM_NOTIFY_QUEUE` 及各模組清單定義表
- Q: 管理畫面放各模組或平台？ → A: **統一於 DP 後台、按模組過濾**；ET 管理者只見 / 只改 `ET_` 前綴（`MODULE=ET`）、DM 只見 / 只改 `DM_`，互不可見；各模組僅保留操作型畫面
- Q: 定義與關聯如何切分？ → A: 「**有哪些**」（分類清單、標籤名稱、func_name、可見對象值、參數值、角色種類 enum）存 `DP_PARAM`；「**誰配誰**」（使用者×角色、使用者×標籤、課程×標籤、文件×分類）與角色能力判定留各模組業務表，引用 DP 的碼
- Q: 是否設「平台管理員」角色？ → A: 否。DP 後台由 ET / DM 管理者進入；共用項（帳號 / 平台級 `DP_` 參數 / 稽核）兩者皆可存取，接受共管取捨
- Q: 權限管理頁指派哪些維度？ → A: 角色與標籤 / 可見對象於**同一頁**指派（ET：角色＋受訓單位標籤；DM：角色＋可見對象 / 單位授權）；兩維度獨立、可各自設定；資料寫入模組表（`ET_USER_ROLE` / `DM_USER_ROLE` / `ET_USER_TAG` / DM 授權表）
- Q: 帳號類信件（密碼重設 / 帳號變更驗證）算參數還是範本？ → A: 改由通知範本 `DP_NOTIFY_TEMPLATE`（`MODULE=DP`）管理；兩管理者皆可編輯主旨 / 內文，但**不可停用 / 刪除**（帳號安全信）

### Session 2026-07-06（模組成立）

- Q: 認證機制採什麼？ → A: 簡單 JWT——驗證帳密後核發一顆 access token（TTL 由參數控制）；**不採 MFA、不採 Refresh Token / 伺服器端 session**；登出＝前端丟棄 token
- Q: 帳號如何啟用？ → A: 〔**自助註冊部分已被 2026-07-21（#56）推翻**：改為「Email 驗證後啟用」——未驗證註冊暫存於 `DP_PENDING_REGISTRATION`、寄驗證信、點連結驗證通過才建 `DP_USER` 並授 ET 學員，詳見 [spec_us2](spec_us2.md) Clarifications 2026-07-21〕；管理者代建之帳號仍直接可用（`MUST_CHANGE_PWD` 首次登入強制改密），不寄開通確認信
- Q: 平台是否做全域 RBAC？ → A: 否。角色能力定義 / 指派資料 / 判定皆由各模組（ET / DM）自理；平台不設 `DP_ROLE` / `DP_MENU` 等全域角色選單表
- Q: 稽核如何集中？ → A: 平台共用一張 `DP_AUDIT_LOG`（append-only）；ET / DM 之**資安稽核事件**亦寫入本表統一查詢；**業務歷程**（DM 文件變更歷程 / 閱讀紀錄、ET 學習 / 作答紀錄）留各模組
- Q: 標準欄位是否含站點？ → A: 否。EDMS 單一組織，標準欄位為 CREATED_USER/DATE、UPDATED_USER/DATE、RES_ID、DELETED（無站點 / 院區維度）
- Q: 密碼重複性如何檢核？ → A: 以 `DP_PWD_HIST` 精簡保存前 N 次密碼雜湊（append-only），禁止與過去 N 次（預設 3）相同

> 其他低影響項採合理預設（不另設問）：密碼雜湊採 bcrypt 或等效不可逆演算法；預設值——登入失敗鎖定 5 次、鎖定 30 分鐘自動解鎖、重設 token 30 分鐘、Email 變更驗證連結 30 分鐘、token TTL＝閒置逾時 15 分鐘、單日換發上限 8 小時、閒置帳號 90 日禁用、密碼到期前 7 天提醒（皆為平台級參數可調）。

## Requirements

### Functional Requirements

各 US 之 Functional Requirements 與 Acceptance Scenarios 置於對應 `spec_usN.md`。本節定義全模組共用之訊息類型與代碼格式。

**訊息類型定義**

各 US 之「系統訊息」表使用以下 5 種類型（與 BS / ET / DM 模組一致）：

| 類型 | 視覺（Bootstrap class）| UI 形式 | 是否需使用者決策 | 用途 |
|------|---------------------|--------|----------------|------|
| 錯誤 | 🔴 紅（`alert-danger`）| 對話框 / Inline 表單驗證 | 否（最多單一【關閉】）| 操作失敗或不允許，使用者必須修正後繼續 |
| 警告 | 🟡 黃（`alert-warning`）| 對話框 / Inline alert | 視情境（對話框需二選一決策、inline 純顯示需注意）| 系統判斷有風險或異常情況 |
| 確認 | ⚪ 中性（Bootstrap modal 白底）| 對話框 | 是（二選一）| 不可逆動作之最終防呆 |
| 成功 | 🟢 綠（`alert-success`）| Inline alert（顯示於按鈕附近）| 否 | 主動操作完成之回饋 |
| 提示 | 🔷 淺藍（`alert-info`）| Inline alert / 列表空白狀態提示 | 否 | 正常流程告知（查詢結果、彙總、引導）|

訊息代碼格式：`DP-MSG-{功能語意碼}-{流水號}`，例 `DP-MSG-LOGIN-001`。DP 無數字型畫面編碼，`{功能語意碼}` 一律採語意碼：`LOGIN`（登入 / 登出）、`REGISTER`（註冊）、`FORGOT`（忘記密碼）、`PROFILE`（個人資料）、`USERS`（使用者管理）、`PARAMS`（系統參數與清單）、`ROLES`（權限管理）、`TEMPLATES`（通知範本）、`AUDIT`（操作記錄）、`MAIL`（發信服務）、`SCHEDULE`（排程）。

| 實體 | 業務定位 |

| 寄件 outbox（DP_EMAIL_LOG）| 平台發信引擎之寄件佇列與結果（僅新增與狀態欄更新，不刪除）；收件人、渲染快照、狀態（PENDING / SENT / FAILED）、重試次數、錯誤訊息、`CALLER_MODULE` |

### 跨模組共用規則

- **帳號共用（SSO）**：ET / DM 共用 `DP_USER`，註冊 / 登入一次即可存取兩系統；登入後由各模組依角色決定可用功能與預設首頁（ET：管理者→ET07 / 教師→ET01 / 學員→ET04；DM 由 DM 決定）。帳號建立當下（自助註冊 / 管理者代建）即自動授予 **ET 學員**（唯一預設角色，透過 ET service 寫入 `ET_USER_ROLE`）；**其他角色（ET 教師 / 管理者、DM 全部四角色〔含閱覽者〕）一律由管理者於權限管理（US7）開通**，未開通者看不到相關功能（2026-07-08 釐清第 3 輪）
- **認證統一由 DP 提供**：登入 / 註冊 / 忘記密碼之畫面與流程皆為 DP（UCDP001–003）；個人資料維護為 DP（UCDP004）；ET / DM 不自設登入 / 個資畫面，僅以連結導向
- **角色分治**：平台不做全域 RBAC、不定義角色能力；角色種類為固定 enum（無新增角色種類），同一使用者可多角色、權限取聯集；判定與 enforce 由各模組於業務操作時執行
- **稽核欄位**：所有資料異動自動記錄建立者 / 異動者、建立時間 / 異動時間；EDMS 單一組織，無站點 / 院區維度
- **軟刪除**：預設軟刪除（`DELETED=1`）、查詢預設過濾；append-only 記錄表（`DP_AUDIT_LOG` / `DP_PWD_HIST` / `DP_SCHEDULE_LOG`）僅含 CREATED 欄位、無 UPDATED / DELETED；`DP_EMAIL_LOG` 為 outbox——僅新增與狀態欄更新（PENDING → SENT / FAILED）、不刪除（含 UPDATED_*、無 DELETED）

### 認證機制（簡單 JWT）

- 驗證帳密通過後核發一顆 **JWT access token**；前端後續請求以 `Authorization: Bearer <token>` 認證
- **閒置自動登出（短 TTL + 活動換發）**：EDMS 認定為含敏感資料系統——token TTL 即閒置逾時（平台級參數，預設 15 分鐘、不得超過 15 分鐘）；前端有操作即**靜默換發**新 token，停止操作逾時 token 自然失效、需重新登入
- **單日換發上限**：自本次登入起算之換發上限（平台級參數，預設 8 小時）；逾上限即使持續操作亦需重新登入
- **不採 MFA、不採 Refresh Token / 伺服器端 session**；登出＝前端丟棄 token 並寫登出稽核
- **登入後導向**：因未登入 / 閒置逾時而被攔到登入頁者（通知信連結、書籤、逾時重登）登入後**返回原目標頁面**（redirect 白名單檢核防 open redirect）；無目標頁面者（含新註冊使用者）顯示**模組入口頁**——ET 入口恆顯示（人人具 ET 學員）、**DM 卡恆顯示：無任何 DM 角色者呈「未開通」鎖定狀態**（引導洽單位主管 / 管理者，點擊不進入）、個人資料維護恆顯示；首次登入顯示**歡迎橫幅**一次（可關閉）；**DP 後台入口僅設於 ET / DM 介面之管理選單**（入口頁不顯示後台入口）
- 除匿名功能（登入 / 註冊 / 忘記密碼）外，所有功能須通過身分驗證後才可存取；帳密加密傳輸（HTTPS）、不以明文傳輸；密碼輸入遮蔽顯示
- **防範自動化程式**：對登入 / 忘記密碼 / 密碼變更端點實施伺服器端速率限制（以來源 IP + 帳號為維度）；不採 CAPTCHA；帳號鎖定機制照舊
- 無站點 / 工作站選擇、無 IP 設備識別（EDMS 單一組織）

### 密碼策略與帳號安全

- **複雜度**：可強制執行密碼複雜度——最小長度（一般使用者至少 8 字元、特權帳號至少 12 字元）、含至少 3 種字元組合（大小寫英文、數字、特殊符號）
- **特權帳號定義**：具 ET / DM 管理者角色之帳號；12 字元門檻於其**下次變更密碼時**適用（被指派管理者角色當下不強制改密碼）
- **重複性**：禁止與過去 N 次（預設 3）相同之密碼；以 `DP_PWD_HIST` 保存前 N 次雜湊檢核
- **效期**：密碼效期限制（最短 1 天、最長 90 日）；**到期後於登入時強制變更密碼方可繼續使用**；到期前由平台排程 `SCHDP001` 寄 Email 提醒（預設到期前 7 天起，天數為平台級參數；範本 `MODULE=DP`「密碼到期提醒」）
- **代建帳號初始密碼**：管理者代建帳號時設定一組初始密碼並標記「**首次登入強制變更**」；系統內「預設密碼」僅此情境（自助註冊由使用者自訂密碼）
- **儲存**：密碼一律以不可逆雜湊儲存（bcrypt 或等效）；加密採公開、國際機構建議且未遭破解之演算法（如 AES / RSA / SHA-2），使用支援之最大金鑰長度
- **密碼變更**：輸入舊密碼 + 新密碼 + 確認新密碼；驗證舊密碼正確、兩次一致、符合複雜度與重複性後更新並寫稽核

### 帳號鎖定與閒置控管

- 連續登入失敗達設定次數（平台級參數，預設 5）→ 自動鎖定帳號；解鎖採**管理者手動解鎖**（US4）或**逾時自動解鎖**（等待時間為參數，預設 30 分鐘）
- 閒置帳號：由平台排程 `SCHDP001` 每日檢核，連續 90 日未登入即自動禁用（管理者可解鎖 / 重新啟用）
- 鎖定 / 解鎖動作一律寫入 `DP_AUDIT_LOG`

### 模組過濾與共用項

- **模組項（互不可見）**：`ET_` / `DM_` 前綴之參數與清單、`MODULE=ET` / `DM` 之範本、各模組角色與標籤指派——ET 管理者只見 / 只改 ET 的、DM 只見 / 只改 DM 的
- **共用項（兩管理者皆可存取）**：帳號管理（`DP_USER`）、平台級無前綴 `DP_` 參數（密碼策略、鎖定、JWT、閒置逾時、發信調校）、操作記錄稽核（含登入等不分模組事件；資安監督需全視野）
- 兼具 ET + DM 管理者身分之使用者，於後台同時可見兩模組之模組項
- **模組側欄可見性（ET / DM 介面內，由各模組 enforce）**：教育訓練功能組人人可見；文件管理功能組僅具任一 DM 角色者可見（未開通＝隱藏）；系統管理後台組僅 ET / DM 管理者可見——與入口頁鎖定卡分工：引導歸入口頁、側欄保持乾淨（2026-07-09 釐清第 4 輪）

### 定義 vs 關聯分層

- 「**有哪些**」的清單定義（分類、func_name、標籤名稱、可見對象值、DM 檢索標籤〔適用模組・文件性質・法規關聯〕、參數值）存 `DP_PARAM`；清單型以一個 `PARAM_ID` 下多筆 `PARAM_KEY` 表達；需鎖定之碼以 `DETAIL_LOCK` 標記（建立後不可改，如文件分類碼）
- 「**誰配誰**」的關聯 / 指派與角色判定留各模組業務表，引用 DP 的碼
- 平台提供**唯讀查詢服務**供各模組讀取自己前綴之定義；讀取後之**業務規則套用歸模組**（分類碼嵌入 DOC_ID、func_name 唯一手冊檢核、標籤「全體」`IS_ALL` 展開與自動邀請等）

### 通知範本與發信引擎

- 範本統一存 `DP_NOTIFY_TEMPLATE`；各模組於業務事件呼叫平台唯一發信服務 `send_email(recipients, template_code, params)`，平台渲染範本 → 寫入 outbox `DP_EMAIL_LOG`（狀態 PENDING）→ 背景 worker 依速率 / 重試參數（平台級）透過外部 SMTP **非同步批次**寄送，更新狀態（SENT / FAILED）；失敗率等系統性告警由 IT 監控機制負責（不做系統內告警通報）
- 各模組**不自持範本表、不自建 outbox**
- 範本停用（啟用停用旗標關閉）→ 該類信件不寄，觸發事件照常運作
- **DP 系統信**（`MODULE=DP`：密碼重設 / 帳號變更驗證 / 密碼到期提醒）為共用項——兩管理者皆可編輯主旨 / 內文，**不可停用 / 刪除**（帳號安全信）
- **站內訊息不在 DP**：站內訊息之儲存表與 UI 留各模組自理；範本之管道欄位（Email / 站內 / 兩者）僅作為該事件**是否寄 Email** 之開關依據，站內文案是否引用範本由模組自行決定
- 範本更新採版本樂觀鎖；版本衝突時提示重載後再存

### 排程引擎

- 平台提供單一排程執行引擎（APScheduler）＋ `DP_SCHEDULE` 註冊表；多實例部署時以 leader 選舉確保只有一個實例觸發（單一實例可簡化）
- 各模組 job（ET 週報 / 提醒 SCHET001 / 002、DM KPI 週報 SCHDM001）於 `DP_SCHEDULE` 登錄、由平台引擎執行；job handler 由各模組提供，需要業務資料時反向 import 模組 service
- 執行歷程記錄於 `DP_SCHEDULE_LOG`（起訖時間、成功 / 失敗、錯誤訊息）；job 失敗不影響其他 job，補跑由各模組視需要處理
- **平台自身排程（`SCHDP001`，每日）**：閒置帳號禁用（連續 90 日未登入）與密碼效期到期前提醒（Email）；亦於 `DP_SCHEDULE` 登錄。outbox 背景 worker 為常駐程序、不登錄於排程表
- **排程總覽（唯讀畫面，共用項）**：DP 後台提供排程總覽——列出 job / cron / 啟停狀態 / 上次執行結果與執行歷程；**不可於 UI 啟停或手動補跑**（啟停由 DB / 部署管理）
- 排程時間等參數存 `DP_PARAM`（前綴分模組），於 DP 後台由各模組管理者維護（US5）

### 稽核（操作記錄）規則

- 至少稽核：更改密碼、登入成功 / 失敗、系統存取成功 / 失敗、帳號建立 / 停用、角色 / 權限修改、密碼重置、系統參數 / 清單 / 範本異動；採單一日誌機制（`DP_AUDIT_LOG`）、格式一致、僅授權使用者可存取
- 日誌項目：使用者 ID（非個資型）、時間至秒、功能 / 資源名稱、執行結果 / 事件描述、網路來源與目的位址、異動前後值（TEXT 存 JSON，非 JSONB，見 [research.md](research.md) §6）
- **append-only**：不得由管理者刪除 / 竄改；須特定授權人員才可異動並留軌跡，以雜湊等方式確保完整性
- 保留：至少 1 年以上；容量失效時自動因應（如覆寫最舊）並留軌跡，容量告警由 IT 監控機制負責；每日備份至不同實體並加密（**備份與告警屬 IT 維運範圍**，不屬系統功能，不入排程表）
- **業務歷程不併入本表**：DM 文件變更歷程（`DM_CHANGE_LOG`）、閱讀紀錄、ET 學習 / 作答 / 擁有者轉讓等為業務資料，留各模組、面向使用者查詢

## Edge Cases（跨 US 共用）

- **新使用者預設權限**：僅 ET 學員（帳號建立當下授予）；未具任何 DM 角色者於模組入口頁見 DM「未開通」鎖定卡（引導洽管理者、點擊不進入）、模組側欄不出現文件管理組，直接存取 DM 亦被拒（模組端 enforce），須由管理者於 US7 開通
- **代建帳號首次登入**：以管理者設定之初始密碼登入後，強制變更密碼方可繼續使用
- **密碼逾效期**：登入驗證通過後導向強制變更密碼頁，未完成變更不得存取其他功能
- **鎖定 / 停用帳號申請忘記密碼**：仍回覆相同訊息（防帳號列舉）；重設成功不解除鎖定 / 停用狀態，仍需管理者處理或逾時解鎖
- **重複申請重設信**：重新申請即產生新 token，舊 token 立即失效（一次性）
- **Email 變更驗證期間**：舊 Email 仍可登入；驗證連結逾時未點 → 變更作廢、舊 Email 維持；新 Email 已被他人註冊 → 阻擋變更
- **管理者操作自己**：管理者不可停用 / 鎖定自己的帳號（自我保護）；管理者間可互相停用，系統不檢核「至少保留 1 名管理者」（0 名時由 IT 經 DB 恢復，比照 ET / DM）
- **停用帳號之既有 token**：帳號停用後，其尚未逾期之 JWT 於下次請求時檢核帳號狀態並拒絕（兩模組同步失效）
- **模組項越權**：ET 管理者以任何方式（含直接呼叫 API）存取 `DM_` 前綴參數 / `MODULE=DM` 範本 / DM 角色指派 → 一律拒絕（伺服器端檢核，非僅前端過濾）
- **鎖定碼修改**：`DETAIL_LOCK` 標記之清單碼（如文件分類碼）建立後不可修改碼值，僅可改名稱 / 停用
- **排程重疊觸發**：前次執行未完成而下次 cron 到期 → 跳過本次並記錄（不重複執行同一 job）
- **發信全部失敗**：寄送失敗依重試上限重試；逾上限標記 FAILED 並留錯誤訊息；失敗率告警由 IT 監控機制追蹤（不做系統內通報）
- **稽核容量**：日誌容量失效時自動因應（如覆寫最舊）並留軌跡；容量告警由 IT 監控機制負責

## Success Criteria

### Measurable Outcomes

- **SC-001**: 使用者以有效帳密登入後取得 JWT（token TTL＝閒置逾時，預設 15 分鐘，操作中自動靜默換發），並可以同一 token 存取 ET 與 DM 授權範圍內功能（單一登入）
- **SC-002**: 連續登入失敗達設定次數（預設 5）時帳號自動鎖定；經管理者解鎖或逾時（預設 30 分鐘）自動解鎖後方可再登入；鎖定 / 解鎖皆留稽核
- **SC-003**: 密碼重設連結為一次性且具時效（預設 30 分鐘）；逾時或已使用即拒絕並提示重新申請；不論帳號是否存在，忘記密碼皆回覆相同訊息
- **SC-004**: 不符複雜度（長度 / 字元組合）或與前 N 次（預設 3）重複之新密碼一律被拒絕；密碼僅以不可逆雜湊儲存；密碼逾效期（預設 90 日）者登入後強制變更方可繼續
- **SC-005**: 連線閒置逾時（≤ 15 分鐘，參數可調）後 token 失效自動登出，需重新驗證身分；持續操作之換發不得超過單日上限（預設 8 小時）
- **SC-006**: 帳號停用後於 ET / DM 兩端同步失效（下次請求即拒絕）；連續 90 日未登入之帳號由 `SCHDP001` 自動禁用
- **SC-007**: ET 管理者於 DP 後台僅見 `ET_` 前綴 / `MODULE=ET` 之模組項，DM 管理者僅見 `DM_`；越權存取（含直接呼叫 API）一律被伺服器端拒絕；共用項兩者皆可存取
- **SC-008**: 各模組透過唯讀查詢服務讀取 `DP_PARAM` 定義後，其業務下拉 / 產碼 / 檢核即時反映最新啟用中清單項；`DETAIL_LOCK` 之碼建立後不可改
- **SC-009**: 各模組呼叫發信服務後，信件經 outbox `DP_EMAIL_LOG` 非同步寄出並留狀態（PENDING / SENT / FAILED 與重試軌跡），不阻塞呼叫方交易
- **SC-010**: 帳號建立 / 停用、角色 / 標籤指派、密碼重置、登入登出、參數 / 範本異動皆寫入 `DP_AUDIT_LOG`（時間至秒、異動前後值），可於 US10 多條件查詢並匯出 CSV；日誌 append-only、保留至少 1 年
- **SC-011**: `DP_SCHEDULE` 註冊之 job 依 cron 準時觸發（多實例僅 leader 觸發一次），每次執行於 `DP_SCHEDULE_LOG` 留起訖與結果；單一 job 失敗不影響其他 job

## Assumptions

- **個資處理**：使用者 Email 與姓名於 EDMS 屬一般個資、不加密儲存（對齊 ET spec 假設）；密碼一律雜湊
- **預設值**：登入失敗鎖定 5 次、鎖定 30 分鐘、重設 token 30 分鐘、Email 變更驗證連結 30 分鐘、token TTL＝閒置逾時 15 分鐘、單日換發上限 8 小時、閒置帳號 90 日、密碼歷史 3 次、密碼到期前 7 天提醒——皆為平台級參數，US5 可調
- **單一組織**：無站點 / 院區 / 工作站維度；無多據點同步
- **部署**：預設單一實例（排程 leader 選舉為多實例時之保障）；管理者介面限制存取來源、系統錯誤僅顯示簡短訊息與代碼、輸入合法性檢查置於伺服器端、金鑰 / 憑證配合週期到期前更換（平台級資安通則，隨各 US 落地）
- **IT 維運範圍**：稽核日誌每日備份（至不同實體並加密）、發信失敗率 / 稽核容量等系統性告警監控——由 IT 以 log / 監控工具負責，不屬系統功能（2026-07-08 釐清第 2 輪）
- **需求編號**：DP 為模組拆分後新增之共用層，無 RFP RQ 編號（TBMS `RQ0.md` 之 RQDP001–063 屬主專案「資訊模組」，不沿用）；需求追蹤以 RQDP.md 功能章節為單位（見〈US → RQDP 章節對照〉）
- **數量 / 長度上限延遲決定**：參數值長度、範本內文長度、outbox 保留期、速率限制門檻等於 `/speckit.plan` 與 data-model 階段依效能與業務情境決定，spec 階段不預先綁定
- **跨模組互動細節**：唯讀參數服務、發信服務、角色指派寫入模組表之介面格式由 [contracts/](contracts/) 規範（platform-services / module-callbacks / ext-dp-email-server）；錯誤碼依 `sti-error-codes` 於實作定義

## Dependencies

- **ET / DM 模組 service**：權限管理（US7）指派結果透過模組 service 寫入模組表；排程 job handler（US11）由各模組提供並反向 import 模組 service
- **外部 eMail Server（SMTP）**：發信引擎（US6）之實際寄送通道；不可用時信件停留 outbox 依重試機制處理
- **ET / DM data-model 對齊**：各模組廢除自持之參數 / 範本 / 佇列定義表、改引用 `DP_PARAM` / `DP_NOTIFY_TEMPLATE` / `DP_EMAIL_LOG`（對齊事項見 `_refs/09-平台模組.md` §4）

## Out of Scope（明確不納入）

- 多站點 / 院區架構（站點主檔、單位維度）
- 全域 RBAC 與功能選單、集中式角色能力定義（角色能力定義 / 指派資料 / 判定皆由各模組自理；DP 僅提供指派畫面）
- 平台管理員角色 / 旗標（DP 後台由 ET / DM 管理者進入）
- MFA 多重驗證、Refresh Token / 伺服器端 session、CAPTCHA
- 站內訊息之儲存與 UI（留各模組自理）
- 系統性告警通報功能（發信失敗率 / 稽核容量等由 IT 監控機制負責）
- 模組開通申請流程（申請單 / 狀態追蹤 / 鎖定卡「申請開通」通知按鈕）——列為未來擴充 backlog（2026-07-09 釐清第 4 輪）
- 資訊設備 / IP 對照、標籤印表機
- 對外系統 API、健檢資料介接、看板、多據點同步
- 血袋編碼 / ISBT128 / 周邊設備等 TBMS 血品業務
- 各模組業務歷程（DM 文件變更 / 閱讀紀錄、ET 學習 / 作答紀錄）之儲存與查詢——留各模組
