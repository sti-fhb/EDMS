## 模組定位

教育訓練文件管理模組（ET）提供血液中心內部人員之線上教育訓練功能：教師建立課程、上傳教材、安排測驗並邀請學員學習；學員透過線上方式完成學習與考核，並可隨時查閱個人學習進度與成績。本模組**獨立於主系統部署**，與文件管理模組（DM）共用同一帳號系統與資料庫；教材文件統一由 DM 模組提供，避免教材與規範文件重複維護。

### 主要角色

| 角色 | 職責 |
|------|------|
| 管理者 | 權限管理（設定其他帳號於 ET 之角色與業務模組對應）|
| 教師 | 課程安排（建立 / 編輯 / 停課課程、章節、教材、測驗；邀請學員；查看學員學習狀況）|
| 學員 | 加入課程、觀看教材、進行線上測驗、查看個人學習進度 |
| 系統自動 | 章節解鎖判定、測驗閱卷、邀請信寄送、模組預設帶入 |

> 角色可重複指派；新使用者首次登入自動授予「學員」角色。

### Key Entities

| 實體 | 業務定位 |
|------|---------|
| 共用使用者主檔（USER）| 與 DM 共用之 user table；含 USER_ID、帳號（Email）、姓名、密碼雜湊；註冊一次即可登入兩系統 |
| 使用者角色（ET_USER_ROLE）| 使用者於 ET 之角色指派（管理者 / 教師 / 學員）；同一使用者可多角色 |
| 使用者業務模組對應（ET_USER_MODULE）| 使用者×業務模組多對多關聯，用於課程預設帶入 |
| 課程主檔（ET_COURSE）| 教師建立之課程；含課程名稱、關聯模組、課程描述、狀態（草稿 / 已發布 / **停課中** / 已停課）、擁有者、邀請碼、首次發布時間（開課日期）|
| 擁有者轉讓紀錄（ET_OWNER_TRANSFER）| 管理者代為轉讓課程擁有者之稽核紀錄；含課程、轉讓前 / 後擁有者、轉讓原因、執行管理者、時間 |
| 章節（ET_CHAPTER）| 課程下之順序容器；學員須依序解鎖 |
| 章節項目（ET_ITEM）| 章節下之教材或測驗項目；含順序、類型（教材 / 測驗）|
| 教材內容（ET_MATERIAL）| 教材項目之三類媒材組合：影片（本地上傳）/ DM 文件引用 / 說明文字 |
| 測驗主檔（ET_QUIZ）| 章節項目為測驗時之測驗設定：及格分數、作答時間限制、重考次數上限 |
| 題目（ET_QUESTION）| 測驗下之題目；含題型（單選 / 多選）、題幹、配分、軟刪除標記 |
| 選項（ET_OPTION）| 題目之選項；含選項文字、是否為正確答案 |
| 選課關聯（ET_ENROLLMENT）| 學員 × 課程多對多關聯；含加入來源（Email 邀請 / 邀請碼 / 模組預設帶入）、加入時間、完課狀態、移除標記 |
| 學習進度（ET_PROGRESS）| 學員於各章節項目之學習進度（影片觀看百分比、文件閱讀標記、上次觀看位置）|
| 影片觀看區段（ET_PROGRESS_INTERVAL）| 學員於某影片教材之已觀看播放區段；每段為一資料列（含起始秒、結束秒），離開頁面時合併重疊 / 鄰近區段；覆蓋率聚合計算 |
| 測驗作答（ET_QUIZ_ATTEMPT_M）| 學員於某測驗之一次作答嘗試；含開始時間、提交時間、得分、是否及格 |
| 作答明細（ET_QUIZ_ATTEMPT_D）| 學員於某次作答之各題作答內容與得分 |
| 邀請紀錄（ET_INVITATION）| Email 邀請紀錄；含課程、Email、寄送時間、狀態（待加入 / 已加入 / 已撤回）|
| 業務模組（ET_MODULE）| 系統內定之業務模組清單（採血 / 成分 / 檢驗 / 供應 / 醫務 / 報表與標籤 / 其他）|
| 系統參數（ET_PARAM）| ET 內部參數表（影片格式 / 影片大小上限 / 章節更新通知模板等）；由 IT 直接於 DB 設定，**不依賴主系統 DP 模組**（ET / DM 獨立於其他模組）|

### 跨模組共用規則

- **帳號共用**：ET 與 DM 共用 user table（USER_ID / 帳號 / 密碼 / 姓名），使用者註冊一次即可登入兩系統；ET 端首次登入自動授予「學員」角色
- **多重角色**：角色（管理者 / 教師 / 學員）可重複指派，權限取聯集；登入後依角色預設首頁導向，多重角色由側邊選單切換
- **教材文件來源**：所有教材引用之文件統一由 DM 模組之「訓練教材」分類提供；DM 端發布新版時 ET 教材自動取得最新版。DM 文件廢止後之 ET 端行為詳見下方「DM 文件廢止之 UI 規則」
- **擁有權判定**：課程之擁有者於建立當下記錄；他人課程僅可閱覽（唯讀），編輯與其他操作按鈕皆不顯示。**例外**：擁有者離職 / 帳號失能時，**管理者可代為轉讓擁有者**至其他教師（破例變更，需於稽核 log 記錄轉讓人 / 接收人 / 時間 / 原因）
- **稽核欄位**：所有資料異動自動記錄建立者 / 異動者、建立時間 / 異動時間
- **獨立部署**：本模組與主系統 TBMS 各業務模組之帳號完全切開；僅與 DM 共用帳號系統

### 強制完成解鎖規則

學員於 US5 章節學習與 US6 線上測驗之解鎖規則：

| 章節包含內容 | 解鎖下一章節條件 |
|-------------|-----------------|
| 僅教材（含影片）| 章節內所有影片需播放覆蓋率達 80% |
| 僅教材（無影片，僅文件 / 說明文字）| 學員開啟該項目即視為已閱讀，純記錄不強制（系統不阻擋下一章節之解鎖）|
| 含測驗 | 上述教材條件達成 AND 測驗達到及格分數 |

**影片 80% 判定方式**：

- **採累計覆蓋率**：學員實際播放過之區段聯集 ÷ 影片總長 ≥ 80% 即達成（非連續播放、非播放時數累加）
- **允許跳轉**：學員可拖動進度條跳轉播放，不強制線性播放
- **重複觀看不加成**：同一區段重複觀看不會多算覆蓋率（區段聯集去重）
- 教材內容可重複觀看；跨章節保持上次觀看位置，學員返回時自動恢復
- 課程於學習中被停課 → 系統顯示「此課程已停止開放」，學員無法繼續學習

> 實作建議：每段播放區間以**獨立資料列**儲存於 `ET_PROGRESS_INTERVAL`（避免 JSON 字串之 race condition、便於以 SQL 直接聚合）；每次播放結束 / 暫停時 `INSERT` 一筆，覆蓋率 = 區段聯集長度 ÷ 影片總長。

**區段儲存規則**：

| 項目 | 規則 |
|------|------|
| 儲存結構 | 獨立 table `ET_PROGRESS_INTERVAL`，每段播放為一資料列（含學員、影片教材 ID、起始秒、結束秒、寫入時間）|
| 區段上限 | 不限（不裁切，避免學員實際觀看資料遺失）|
| Normalize 時機 | **學員離開頁面時合併重疊 / 鄰近區段**（前端發送 `unload` 事件 → 後端執行合併：SELECT all → 排序 → 合併 → DELETE all → INSERT 合併後結果）；播放中不即時合併以避免高頻寫入 |
| 覆蓋率計算 | 即時聚合：SUM(END_SEC − START_SEC) ÷ VIDEO_DURATION（normalize 前後皆可正確計算，因聚合方法相同）|

> 學員未正常離開（瀏覽器強制關閉、斷網）時 normalize 不執行，但區段紀錄已 INSERT 完成，覆蓋率計算仍正確；下次學員再次開啟並離開時補做 normalize。

### 多選題評分規則

採部分計分（Moodle 預設）：

```
該題得分 = max(0, (答對正確選項數 − 誤選數) ÷ 應選正確數 × 該題配分)
```

- 每答對一個正確選項加分
- 每選錯一個錯誤選項扣分
- 最低為 0 分（不會出現負分）
- 單選題仍為全有全無（答對得滿分、答錯 0 分）

**邊界規則**：

- **題目建立時必填檢核**：系統強制檢核「至少 1 個正確選項」；未滿足者阻擋儲存並提示「請至少指定 1 個正確選項」。「全為錯誤選項」題型不允許（避免公式分母為 0）
- **完全未作答之多選題**：學員提交時若該題完全未勾選任何選項，視為 0 分（不另保留空白讓教師審閱）

### 測驗作答機制（題目與選項洗牌）

| 範圍 | 行為 |
|------|------|
| 同一作答 session 內 | 題目順序與選項順序**保持一致**（學員跳題、暫存後回來，順序不變）|
| 重考時 | **重新洗牌**（題目順序與選項順序皆重新洗，避免學員記答案位置）|
| 多選題之選項順序 | **隨題目一同洗牌**（單選題之選項亦比照）|
| 順序快照 | DB 於 `ET_QUIZ_ATTEMPT_M` 存「該次作答之題目順序與選項順序快照」，供答題明細顯示與爭議釐清 |

> 實作建議：`ET_QUIZ_ATTEMPT_M` 新增 `QUESTION_ORDER` 與 `OPTION_ORDER`（每題之選項順序對應陣列）；學員作答中即依快照渲染題序，提交後依快照產出答題明細。

### 測驗設定邊界值

| 欄位 | 型別 | 值域 | 特殊值 |
|------|------|------|--------|
| 及格分數 | INT | 0–100 | 預設 80 |
| 作答時間限制（分鐘）| INT | ≥ 0 | **0 = 不允許作答**（測驗會被視為未啟用，學員端不顯示）|
| 重考次數上限 | INT | ≥ -1 | **0 = 不允許重考**（僅可作答 1 次）；**-1 = 不限重考次數** |

> **作答時間限制 = 0** 與「不限時間」不同：欲設不限時間需另設大數（plan 階段定義；不建議用 0）。
> **學員端 UI 行為**：作答時間限制 = 0 之測驗於章節學習頁直接隱藏不顯示，且不阻擋下一章節解鎖（視同無測驗）。

### 閱卷後正確答案顯示規則

學員提交測驗並完成自動閱卷後，於答題明細頁**強制顯示正確答案**：

- 所有測驗皆強制顯示，無教師設定可關閉之選項（避免題庫不一致或教師設定遺漏）
- 學員「未及格但重考次數未用完」時**仍顯示**正確答案；重考時依「測驗作答機制」重新洗牌之題目順序與選項順序作答
- 顯示範圍：題幹原文 + 所有選項（含正確答案標示）+ 學員當次選擇 + 該題得分

> 即便題庫外流風險存在，仍以「學習導向」為設計優先；未來若需限制可於 plan 階段討論加入教師設定。

### 軟刪除分流規則

刪除章節 / 題目時，**本體採軟刪除**（DELETED=1，前台不顯示但 DB 保留供稽核）；**學員學習紀錄與測驗成績連同 hard delete**（避免孤兒紀錄，且完課率以當前章節數為分母重算）：

| 刪除動作 | 軟刪除對象（DB 保留）| Hard delete 對象（DB 移除）|
|---------|---------------------|--------------------------|
| 刪除章節 | 章節本體、章節下教材與測驗本體 | 學員於本章節之學習紀錄與測驗成績 |
| 刪除題目 | 題目本體 | 學員於該題之作答紀錄與得分 |

> **「移除學員」不同於上述**：學員 ET_ENROLLMENT 標記為已移除（前台不顯示），但學習歷史紀錄完整保留；移除後不計入完課率統計。
>
> **學員無法主動退出課程**：學員端 ET04 我的課程頁僅顯示課程清單，無「退出課程」按鈕；如需退出，由教師於 US9 學員管理頁執行「移除學員」。

### 並發處理規則（教師動作 vs. 學員進行中）

當教師端執行「停課 / 移除學員 / 編輯課程內容」等動作時，若有學員正在學習或作答中，採用以下原則：

#### 原則 1：Attempt Snapshot（作答即凍結規則）

學員一旦點「開始測驗」進入答題介面，系統為該次 `ET_QUIZ_ATTEMPT_M` 拍一份規則快照：題目與選項本體、題目與選項順序、總題數、配分、及格分數、作答時間限制。在該 attempt 結束（提交 / timeout 自動提交）之前，教師端任何修改皆**不影響本次 attempt**。新開啟之 attempt 才套用最新版規則。

#### 原則 2：Graceful Termination（允許完成本次學習動作）

教師端「較大動作」（停課 / 移除學員）採「允許完成 → 之後生效」之延遲處置，避免吞分爭議。

#### 各場景處理

| 教師動作 | 學員狀態 | 系統行為 |
|---------|---------|---------|
| **課程停課（US11）** | 無人作答中 | 直接停課，學員端顯示「此課程已停止開放」 |
| | 有學員作答中 | 停課按鈕跳警告「目前有 N 位學員作答中，停課後將於其等提交後生效。確定停課？」；確認後課程進入 `PENDING_CLOSE` 過渡狀態：新學員無法加入、無法開新 attempt、已加入學員可閱覽但不可開新 attempt，仍允許作答中學員完成本次（含 timeout 自動提交）；全部提交後系統自動將 `PENDING_CLOSE` → `CLOSED` |
| **移除學員（US9）** | 無作答中 attempt | 立即標記移除，學員 next request 被導至「您已被該課程移除」訊息頁；ET_ENROLLMENT 之 REMOVED_AT 寫入 |
| | 有作答中 attempt | 移除按鈕跳警告「該學員作答中，移除後其當前 attempt 將保留並計入歷史。確定移除？」；確認後立即標記移除；學員當前 attempt 沿用快照繼續，提交後正常閱卷並計入歷史紀錄；學員 next page navigation 被導至「您已被該課程移除」訊息頁 |
| **刪除章節 / 題目** | 學員觀看 / 作答該內容 | 學員端顯示「此內容已刪除」訊息；作答中之 attempt 沿用 snapshot 完成本次 |
| **修改章節 / 教材 / 題目本體** | 學員學習 / 作答中 | 已開啟之 attempt **沿用 snapshot**（題幹、選項、配分、順序皆凍結）；新開啟之 attempt 套用最新版（含重新洗牌）；學員學習中觀看之教材則於下次重新整理時取得最新版 |

> **PENDING_CLOSE 狀態**：作為「已發布 → 已停課」之過渡，DB 上以獨立狀態值區分。教師端 UI 於該狀態下顯示「停課中（等待 N 位學員提交）」提示；無人作答後排程或事件觸發自動轉 CLOSED。
>
> **`ET_QUIZ_ATTEMPT_M` snapshot 欄位**：除 `QUESTION_ORDER` / `OPTION_ORDER` 外，須額外保存題目 + 選項 + 配分 + 及格分數 + 作答時間限制之版本快照（具體欄位於 data-model 階段定義）。

### 多裝置同時編輯之衝突處理（樂觀鎖）

同一教師（owner）於 A、B 兩裝置同時開啟 ET02 編輯同一課程時，採**樂觀鎖**：

- 每筆課程 / 章節 / 教材 / 測驗 / 題目本體於 DB 維護一 `VERSION` 欄位（每次寫入即 `VERSION = VERSION + 1`）
- 教師於 A 裝置開啟編輯時，前端記錄當下 `VERSION`；儲存時 server 端檢核傳入 `VERSION` 是否等於 DB 當下 `VERSION`：
  - **相等** → 寫入成功，`VERSION` 遞增
  - **不等** → 拒絕寫入，回傳「**內容已被其他裝置變更**，請重新整理後再儲存」提示；學員 / 教師需自行決定保留哪版（系統不自動合併）
- 此檢核覆蓋粒度為**單一實體**（如某課程之單筆章節），非整個課程；同一課程不同章節之同時編輯不互相衝突

### 管理者角色之最後一個保護

spec.md 已規範：「**當前登入之管理者無法停用自己之管理者角色**」（自我保護）。其他保護機制為：

| 情境 | 系統行為 |
|------|---------|
| 管理者 A 停用其他管理者 B 之角色 | **允許**（不檢核「停用後系統是否仍有啟用中管理者」）|
| 多名管理者依序互相停用，最終 0 名管理者 | **允許**（不阻擋）；此情境恢復需透過系統初始化流程之 DB 直接寫入新管理者 |

> **設計決策**：不增「至少保留 1 名啟用中管理者」之系統檢核。若意外造成 0 管理者狀態，由 IT 透過 DB 寫入恢復（per「系統初始化（第一個管理者）」之同等流程）；此情境屬極少數，不值得增加系統檢核複雜度。

### DM 文件廢止之 UI 規則

當 DM 端將某份教材文件標記為「廢止」後，ET 端對該文件之引用採以下處置：

| 端點 | 行為 |
|------|------|
| **教師端編輯頁（ET02）** | 教材編輯視窗於該引用旁顯示「**此文件已廢止**」警告標籤 |
| **教師端發布檢核** | 發布按鈕點擊時，若課程仍引用廢止文件 → **阻擋發布**並提示「請先移除廢止文件之引用後方可發布」 |
| **教師端移除引用** | 教師須**開啟該教材編輯視窗逐筆刪除廢止文件之引用**（系統不主動清除、不提供批次移除）；移除後重新發布 |
| **學員端章節學習頁（ET05）** | 於該文件項目旁顯示「**此文件已廢止**」標籤，學員仍可閱讀廢止前最後版本（per 跨模組共用規則） |

> 教師建立**新教材時**不可從下拉選單選到已廢止文件（DM「訓練教材」分類之下拉清單僅顯示有效文件）；既有已發布課程之既有引用則依上述規則處理。

### 邀請與加入課程

| 加入來源 | 觸發時機 | 識別 | 備註 |
|---------|---------|------|------|
| Email 邀請 | 教師寄發邀請信，學員點擊邀請連結 | 邀請連結內含 token | 學員可於 ET_INVITATION 追蹤狀態 |
| 邀請碼 | 課程發布時系統自動產生 8 碼唯一邀請碼；學員於 ET04 輸入加入 | **純數字 8 碼**（系統產生，唯一）| 邀請碼於課程停課後失效；教師**不可手動指定**、**不提供重新產生功能**（避免外洩管理複雜化）|
| 模組預設帶入 | 教師建立課程指定「關聯模組」後系統自動加入該模組所屬人員 | 由 ET_USER_MODULE 對應決定 | 學員加入來源標記為「模組預設」|

**業務模組對應變更後對既有 / 新課程之影響**：

| 變更類型 | 對既有課程之學員名單影響 | 對之後新建課程之邀請 |
|---------|--------------------|---------------------|
| **移除**使用者 × 業務模組對應 | **不變動**（已加入之學員名單保留；學員可繼續學習）| 之後新建之該模組課程，**不會**自動邀請該使用者 |
| **新增**使用者 × 業務模組對應 | **自動加入過去該模組所有已發布課程**（既有課程之學員名單追加）| 之後新建之該模組課程，**會**自動邀請該使用者 |

> 學員之主動退場僅能由教師於 US9 學員管理頁執行「移除學員」；模組對應變更**不會自動移除既有學員**。

### 重考次數重置規則（US9）

教師於學員清單頁可對個別學員執行「重置重考次數」：

- **重置範圍**：以**特定測驗**為單位（非整個課程之所有測驗一次重置）；教師於該測驗列點擊重置按鈕
- **重置動作**：將該學員於該測驗之「已用重考次數」歸 0，即恢復至滿配額（重考次數上限）狀態
- **可重置時機**：僅當學員於該測驗之「已用重考次數已達上限且尚未及格」時，重置按鈕方可點擊；其他狀態（仍有重考次數 / 已及格）下按鈕不顯示或不可點擊
- **不存在的情境**：因觸發條件已限制為「次數歸 0 時才可重置」，故不會發生「作答中重置」之競態狀況

### 重考時機（無冷卻）

學員測驗未及格、且剩餘重考次數 > 0 時，可**立即**點「重新作答」進入下一次 attempt：

- 系統**不設冷卻時間**（cooldown）；學員提交後即可立刻重考
- 重考開啟新 `ET_QUIZ_ATTEMPT_M` 並依「測驗作答機制」重新洗牌題目與選項順序
- 若需限制重考節奏，可透過「重考次數上限」（`ET_QUIZ.MAX_RETRY`）控制，不另以時間限制

### 完課定義與完課率計算（US9）

| 名詞 | 定義 |
|------|------|
| 完課 | 學員於該課程**所有章節**之教材與測驗皆達成解鎖條件，且**所有測驗皆達及格分數** |
| 完課狀態 | 已完成 / 進行中 / 未開始（依學員之 ET_PROGRESS 即時判定）|
| 完課率 | `已完課學員數 ÷ 已加入學員數 × 100%`；**分母不含已移除學員**（per UC「移除後不計入完課率統計」）|
| 平均成績 | 該學員於該課程**所有測驗**之得分平均；每測驗以**該學員之最高分**為基礎（per RQ「以最高分為結業成績」），再對所有測驗取算術平均。**未作答之測驗排除於計算**（不視為 0 分；分母為已作答測驗數），學員完課前該欄位之意義為「目前為止之平均」|

> 學員完課狀態於每次學習動作 / 測驗提交後即時重算；不依賴排程批次更新。

### 已發布課程之章節異動規則

教師於已發布狀態下仍可繼續編輯課程內容（新增章節、新增教材 / 測驗、修改既有內容），對既有學員之影響如下：

| 異動類型 | 對既有學員之影響 |
|---------|------------------|
| 新增章節 | 已完課學員（狀態為「已完成」）**自動回退為「進行中」**；該學員需重新學習新章節並通過測驗方可重新達成完課 |
| 新增教材 / 測驗（既有章節下）| 該章節下之既有學員需補完新教材 / 測驗，未完成者於該章節進度回退；已完課學員一併回退為「進行中」|
| 修改既有教材 / 測驗（不改本體 ID）| 既有學員之學習紀錄與成績保留，但需重新觀看 / 作答否則該章節進度顯示為「需更新」（具體 UI 於 plan 階段定義）|
| 刪除章節 / 題目 | 依「軟刪除分流規則」處理（章節 / 題目本體軟刪除，學員紀錄 hard delete）|

**通知機制**：

- 教師於已發布課程新增章節後，系統**每次新增即自動寄信**通知所有已加入學員（包含已完課學員），告知課程內容有更新
- 信件內容：課程名稱、新增章節名稱、進入學習連結
- 通知與 US8 邀請信走相同之 Email Server 介接
- 系統**不採 debounce / 合併寄送**亦**不提供學員端通知偏好設定**；實務上教師於初次發布前完成所有章節編排（發布時透過 US8 邀請信通知學員加入），後續對已發布課程之章節新增屬少數情境，每次寄信不易造成困擾

### 系統初始化（第一個管理者）

- **首次部署時**，由 IT 人員透過 DB 直接寫入第一筆管理者帳號（含帳號 Email、姓名、密碼雜湊、ET_USER_ROLE.ROLE=管理者）
- 後續所有管理者帳號之新增 / 變更皆透過 US1 權限管理畫面執行；系統不再提供 DB 寫入以外之初始化介面
- 與部署 / 維運文件約定：第一筆管理者帳號之 Email 與初始密碼於部署完成後由系統管理員另行通知

### 帳號（Email）變更流程

使用者於 US10 變更帳號（Email）時，採「**雙信箱共存 + 新 Email 驗證後切換**」之延遲生效機制：

| 階段 | 狀態 | 認證行為 |
|------|------|---------|
| 1. 提交變更 | 寫入 `USER.EMAIL_PENDING_CHANGE` + `EMAIL_PENDING_TOKEN` + `EMAIL_PENDING_EXPIRES_AT`（30 分鐘後） | 舊 Email **仍可正常登入**（變更尚未生效）|
| 2. 系統寄驗證信至**新 Email** | 同上 | 同上 |
| 3a. 學員 30 分鐘內點驗證連結 | `USER.EMAIL` 更新為新值；清除 PENDING 欄位 | 變更生效；學員當前 session 強制登出，**須以新 Email 重新登入** |
| 3b. 學員未點驗證連結 / 逾 30 分鐘 | PENDING 欄位於下次登入或排程清理時失效 | 舊 Email **永遠維持有效**；變更請求視為作廢 |
| 4. 學員於 PENDING 期間再次提交新變更 | 舊 PENDING 紀錄被**取代**，新驗證連結重新計時 30 分鐘 | 舊 Email 仍可登入 |

> 系統不主動「逾期回滾」之 cron；PENDING 欄位於下次與該帳號相關之認證 / 變更動作時即時檢核並清理。
>
> 對應實體：`USER` 需新增 `EMAIL_PENDING_CHANGE`、`EMAIL_PENDING_TOKEN`、`EMAIL_PENDING_EXPIRES_AT` 三欄（data-model 階段定義）。

## Assumptions

- **SSO 範圍與權責切分**：本模組以 SSO（Single Sign-On）方式與 DM 模組共用 user table（USER_ID / 帳號 / 密碼 / 姓名），使用者註冊一次即可登入兩系統；但 ET / DM 各自管理自己的角色與業務模組對應（角色不共用、功能權限不共用）。SSO 為登入方式而非獨立模組，無 SS 模組編碼。與主系統 TBMS 各業務模組之帳號完全切開（per 2026-04-28 設計決策）
- **業務模組清單**：採血 / 成分 / 檢驗 / 供應 / 醫務 / 報表與標籤 / 其他七類，由系統內定，不開放管理者新增
- **影片儲存**：影片採本地上傳，不串接外部影音平台（YouTube、Vimeo 等）。可接受之格式（預設 mp4 / webm）與單檔大小上限（預設 500 MB）由 **ET 內部系統參數**（`ET_PARAM`）控制，IT 可直接於 DB 調整；ET / DM 獨立於主系統各業務模組，**不依賴 DP 模組之參數機制**
- **DM 文件分類**：教師可選用之文件限 DM「訓練教材」分類；其他分類不出現於下拉選項
- **完課定義**：學員完成課程中所有章節之學習與測驗即視為完課；完課率公式已於 spec §完課定義與完課率計算 定義
- **個資處理**：使用者 Email 與姓名為一般個資，不需加密儲存；密碼採雜湊儲存
- **跨模組互動細節**：請求 / 回應格式、錯誤碼由 `specs/et/contracts/` 規範（待 `/speckit.plan` 階段產出）
- **數量 / 長度上限延遲決定**：課程 / 章節 / 教材 / 測驗 / 題目之**數量上限**，以及課程描述 / 章節名稱 / 教材說明文字 / 測驗名稱 / 學員人數等**長度與容量上限**，於 `/speckit.plan` 與 data-model 階段依效能與業務情境綜合決定，spec 階段不預先綁定
