# 研究紀錄：教育訓練文件管理模組（Education & Training）

**日期**: 2026-06-09（2026-07-02 依客戶 6 項需求變更增修 #7 / #13 / #14、新增 #17～#22）
**規格**: [spec.md](spec.md)

---

## 決策紀錄

### 1. ET 與 DM 共用 user table 但角色各自管理（SSO）

**決策**: ET 與 DM 透過 SSO（Single Sign-On）共用同一 user table（USER_ID / 帳號 / 密碼 / 姓名），使用者註冊一次即可登入兩系統；但兩模組**各自管理自己的角色與受訓單位標籤**，互不共用。SSO 為登入方式而非獨立模組，不另設 SS 模組編碼。與主系統 TBMS 各業務模組之帳號完全切開。

**理由**:
- ET 與 DM 目標族群相同（血液中心內部人員），帳號重複維護徒增管理負擔
- 兩模組業務性質不同（教育訓練 vs. 文件管理），角色與權限體系不互通；強行共用會耦合兩系統
- 與 TBMS 主系統各業務模組之使用者群不同（軍人 vs. 護理師 / 醫師），不可共用帳號池

**排除方案**:
- 完全獨立的帳號系統：使用者重複註冊兩次、密碼變更需同步兩處，UX 差
- 角色亦共用：兩模組業務不同，角色語意衝突（ET 教師 vs. DM 文件審核者）
- 整合進主系統 DP 模組：兩模組獨立部署，不應依賴主系統認證

---

### 2. 教材文件統一由 DM 模組提供

**決策**: 課程之教材所引用之文件統一由 DM 模組「訓練教材」分類提供；教師於 ET02 編輯教材時從 DM 下拉選取既有文件建立關聯；DM 端發布新版本時 ET 教材自動取得最新版；DM 文件廢止後 ET 教材於編輯頁顯示警告並阻擋發布。

**理由**:
- 避免訓練教材與規範文件兩處維護版本不一致
- DM 為唯一文件版本控制來源，已有完整版本 / 簽核 / 廢止流程
- ET 不另設文件儲存空間，降低系統複雜度

**排除方案**:
- ET 自管文件：與 DM 重複功能，版本管理散佈兩處
- 上傳檔案不關聯 DM：失去 DM 之版本控制與廢止機制
- 僅允許 DM 文件之 URL 連結：學員需離站閱讀，無法整合進章節學習頁

---

### 3. 影片本地上傳（不串接外部影音平台）

**決策**: 教材影片採本地上傳（mp4 / webm，單檔上限 500 MB，由 `ET_PARAM` 控制），HTML5 video player 播放，**不串接 YouTube / Vimeo 等外部影音平台**。

**理由**:
- 內部訓練教材有保密性考量，不適合上傳至公開平台
- 外部平台依賴網路可用性、可能廣告插入、API 變動風險
- 學員端不需安裝外部播放器或外掛，使用 HTML5 即可

**排除方案**:
- YouTube / Vimeo 嵌入：保密與廣告問題
- 公司私有 OSS / 串流伺服器：增加基礎建設依賴；本案規模未必需要
- 不支援影片：無法滿足 RQET001 影片教材需求

---

### 4. 章節 / 題目本體軟刪除 + 學員紀錄 hard delete

**決策**: 教師刪除章節 / 題目時，**章節 / 題目本體採軟刪除**（DELETED=1，前台不顯示但 DB 保留供稽核）；**學員於該章節 / 題目之學習紀錄與測驗成績連同 hard delete**（避免孤兒紀錄）。「移除學員」不同：學員 ET_ENROLLMENT 標記為已移除，但學習歷史紀錄完整保留。

**理由**:
- 章節 / 題目本體保留供稽核（誰刪了什麼、何時刪），亦避免 FK 連動爆炸
- 學員紀錄依賴於章節 / 題目存在才有意義，孤兒化後無 UI 可呈現
- 完課率以當前章節數為分母自動重算，學員紀錄保留反而干擾分母
- 「移除學員」保留歷史是因稽核需求（教師曾邀請過誰），與內容刪除不同性質

**排除方案**:
- 三者皆軟刪除：學員紀錄孤兒化，無法顯示
- 三者皆 hard delete：失去稽核能力
- 章節 / 題目 hard delete + 學員紀錄保留：學員紀錄找不到對應章節，UI 崩潰

### 5. 多選題採 Moodle 部分計分

**決策**: 多選題評分採 Moodle LMS 標準之部分計分公式 `max(0, (對 − 誤) ÷ 應選 × 配分)`：每答對一個正確選項加分、每選錯一個錯誤選項扣分，最低 0 分。單選題仍為全有全無。題目建立時強制至少 1 個正確選項；學員完全未作答之多選題視為 0 分。

**理由**:
- 「全有全無」（任一誤選 = 0 分）過嚴；學員可能因小失誤吞分產生質疑
- 「比例給分（無扣分）」過鬆；學員可能全部勾選即可拿大部分分數
- Moodle 公式為業界主流（LMS 標準），達到「鼓勵正確 + 懲罰亂選」之平衡
- 公式之 max 保護避免出現負分

**邊界**:
- 「全為錯誤選項」題（如「以下何者**錯誤**？」）會導致分母為 0，故題目建立時強制檢核「至少 1 個正確選項」
- 學員完全不選之多選題：視為 0 分（不另保留空白讓教師審閱），UX 較清楚

**排除方案**:
- 全有全無：學員質疑「我答對 4 題僅誤選 1 題仍得 0」，公平性差
- 比例給分（無扣分）：學員全勾即可拿大部分分數，失去評量意義
- 教師自訂評分規則：複雜度高、教師教學負擔大

### 6. Attempt Snapshot（並發控制：作答即凍結規則）

**決策**: 學員一旦點「開始測驗」進入答題介面，系統為該次 `ET_QUIZ_ATTEMPT_M` 拍一份規則快照（題目 + 選項 + 配分 + 及格分數 + 作答時間限制 + 題目順序 + 選項順序）。在該 attempt 結束（提交 / timeout 自動提交）之前，教師端任何修改皆不影響本次 attempt；新開啟之 attempt 才套用最新版。

**理由**:
- 業界 LMS（Moodle / Canvas / Coursera）標準做法
- 學員作答中若題目被修改 / 重新洗牌，當前進度將被打斷或失效，體驗極差
- 學員提交後若分數因規則變動而不一致，會引發吞分爭議
- 快照儲存可供後續答題明細顯示與爭議釐清；具體 table 為 `ET_QUIZ_ATTEMPT_M` 與 `ET_QUIZ_ATTEMPT_D`

**排除方案**:
- 教師端修改即時生效：學員作答中體驗崩潰，吞分爭議
- 教師端修改後阻擋學員提交：學員無辜被中斷
- 不存快照只存版本號：無法在事後重建作答時之題目順序與內容

---

### 7. Graceful Termination（關閉課程 / 移除學員之延遲處置）（2026-07-02 修訂）

**決策**: 教師端 / 系統之「關閉課程」與「移除學員」大動作，採「允許完成本次 → 之後生效」之延遲處置：
- **關閉課程**（到期自動或手動）：課程狀態**立即**轉 `CLOSED`；關閉當下**作答中之 attempt 沿用快照允許完成並計分**（含 timeout 自動提交），提交後不可再開新 attempt；學員端其餘功能即時轉唯讀
- **移除學員**：若該學員作答中，當前 attempt 沿用快照完成並計入歷史；學員 next page navigation 被導至「您已被該課程移除」訊息頁

> **2026-07-02 修訂**：原 `PENDING_CLOSE` 過渡狀態**移除**。原設計因「停課為不可逆終態」需過渡保護；改版後關閉為**可逆**（可再開課），且作答中 attempt 本就受 Attempt Snapshot 保護可獨立完成，過渡狀態失去存在必要，狀態機簡化為 DRAFT → PUBLISHED ⇄ CLOSED。

**理由**:
- Moodle / Canvas / Coursera 等業界 LMS 主流做法
- 強制中斷學員作答將導致資料丟失、吞分爭議
- 關閉已可逆，無需終態前之過渡緩衝；快照機制已保證作答中學員不受影響

**排除方案**:
- 立即強制終止（學員作答中直接吞分）：吞分爭議、體驗差
- 保留 PENDING_CLOSE：關閉可逆後無存在必要，徒增狀態機複雜度
- 完全不允許學員作答中關閉 / 移除：教師失去管理彈性

### 8. 影片觀看區段獨立 table（非 JSON 欄位）

**決策**: 學員影片觀看區段以**獨立 table** `ET_PROGRESS_INTERVAL` 儲存，每段播放為一資料列（含起始秒、結束秒、寫入時間）；播放中每次暫停 / 結束 `INSERT` 一筆；學員離開頁面時執行 normalize（SELECT 全部 → 排序 → 合併重疊 / 鄰近區段 → DELETE 全部 → INSERT 合併後結果）。

**理由**:
- CLAUDE.md 規範 PostgreSQL 限用標準型別，**未列 JSONB**
- 若以 TEXT 儲存 JSON 字串，每次播放動作需「讀 → parse → 修改 → 序列化 → 寫」，多裝置 / 多 tab 易產生 race condition
- 獨立 table `INSERT` 為原子操作，無 race
- SQL 直接聚合（SUM(END_SEC - START_SEC)）計算覆蓋率，比應用層 parse JSON 簡單
- 後續可分析「哪段最常被反覆觀看」等學習行為（若未來需要）

**排除方案**:
- TEXT JSON 字串：race condition、parse 成本、查詢困難
- JSONB（PostgreSQL 原生 JSON 型別）：違反 CLAUDE.md 型別限制
- 純數值百分比欄位：失去區段資訊，無法支援「累計覆蓋率」之嚴格判定

---

### 9. 邀請碼純數字 8 碼、不可重新產生 / 手動指定

**決策**: 課程發布時系統自動產生 8 碼純數字唯一邀請碼，寫入 `ET_COURSE.INVITATION_CODE` 後**永久不可變更**；教師不可手動指定、不提供重新產生功能。

**理由**:
- 純數字較易於 QR Code / 口頭傳遞 / 手機輸入
- 8 碼可提供 10^8 = 1 億組合，對血液中心規模足夠
- 「不可重新產生」簡化邀請碼管理；若擔心外洩可透過停課重建課程處理
- 「不可手動指定」避免易記字串導致重複或猜測

**排除方案**:
- 英數混合：輸入易出錯（O vs. 0、l vs. 1）
- 教師可重新產生：管理複雜（舊碼是否失效、學員舊邀請連結處理）
- 教師可手動指定：易重複、易被猜測

### 10. 帳號（Email）變更採雙信箱共存模式

**決策**: 使用者於 ET08 變更 Email 時，系統寫入 `USER.EMAIL_PENDING_CHANGE` + token + 30 分鐘有效期，寄驗證信至**新 Email**；舊 Email 變更期間**仍可正常登入**；學員 30 分鐘內點驗證連結後 USER.EMAIL 更新並強制登出；未點則變更請求作廢、舊 Email 永遠維持有效。同使用者再次提交新請求**取代**前次。

**理由**:
- 避免「變更後驗證未完成即無法登入舊 Email」之認證真空
- 30 分鐘 TTL 與密碼重設連結同（per `ET_PARAM.PASSWORD_RESET_TTL_MIN`），UX 一致
- 「取代」設計避免使用者誤打 Email 後不知如何救回
- 不採排程自動回滾（系統不需 cron）：PENDING 欄位於下次認證 / 變更時即時檢核並清理

**排除方案**:
- 變更後立即將 USER.EMAIL 改為新值，但要求驗證後才可登入：若驗證失敗，使用者徹底無法登入
- 不設驗證流程：使用者誤打 Email 後無法救回
- 排程定時回滾未驗證之 PENDING：系統需 cron，不符「不依賴 DP / 不引入額外基礎建設」原則

### 11. 課程擁有者轉讓由管理者代為執行

**決策**: 課程擁有者於建立當下記錄；**本擁有者欄位永久不可變更**為原則，但**例外**：擁有者離職 / 帳號失能時，**管理者可代為轉讓擁有者**至其他教師（破例變更，需於 `ET_OWNER_TRANSFER` 稽核紀錄記錄轉讓人 / 接收人 / 時間 / 原因）。一般教師不可主動轉讓。

**理由**:
- 軍方環境人員調動頻繁，「擁有者永久不可變」會造成課程鎖死
- 由管理者代為執行（非 owner 自選接收人）避免「擁有者放棄」造成的混亂
- 稽核 log 紀錄可追溯轉讓事實，符合軍方稽核要求
- 一般教師不可主動轉讓避免擁有權混亂

**排除方案**:
- 完全不可變更：擁有者離職後課程鎖死，需重建同名課程
- 教師自行轉讓：可能濫用、缺乏管控
- 「協同擁有者」概念：增加 data model 複雜度（多對多），但 ET 規模未必需要

---

### 12. 不設「至少保留 1 個管理者」之系統檢核

**決策**: 管理者可停用其他管理者之角色；不檢核「停用後系統是否仍有啟用中管理者」。若意外造成 0 管理者狀態，由 IT 透過 DB 寫入恢復（per 系統初始化第一個管理者之同等流程）。但**當前登入之管理者無法停用自己之管理者角色**（自我保護仍保留）。

**理由**:
- 「0 管理者」情境極少（管理者連續互相停用），多檢核之收益低
- 此情境恢復成本不高（IT 寫 DB 一筆 SQL）；增加系統複雜度收益不划算
- 自我保護仍保留以避免單一管理者誤操作鎖自己

**排除方案**:
- 強制檢核「至少 1 個管理者」：每次角色變更皆需 query 總管理者數，增加系統複雜度
- 完全無自我保護：單一管理者誤操作即鎖死，需 IT 介入

---

### 13. ET_PARAM 不依賴主系統 DP 模組

**決策**: ET 內部參數（影片格式 / 大小上限 / 倍速上限 / TTL / 邀請碼長度 / 排程時間 / 加急提醒天數等）由 ET 自管之 `ET_PARAM` table 儲存，IT 直接於 DB 設定；**不呼叫 DP 模組之參數機制**。（信件範本 2026-07-02 移至 `ET_NOTIFY_TEMPLATE`，由管理者於 UI 維護。）

**理由**:
- ET / DM 獨立於主系統 TBMS 部署，不應引入 DP 依賴
- 部署 / 運維上 ET / DM 可單獨啟動，不需等 DP 模組就緒
- 參數變更頻率低（影片格式、TTL 等），IT 直接於 DB 設定即可，無需 UI

**排除方案**:
- 透過 DP 模組統一管理：違反「ET / DM 獨立部署」設計
- ET 端建 UI 管理參數：管理者通常不需調這些參數，建 UI 投資回報低
- Hard code 於程式碼：未來變更需重新部署

---

### 14. 影片觀看 80% 採累計覆蓋率（允許跳轉、允許倍速、重複觀看不加成）（2026-07-02 客戶確認）

**決策**: 「影片需播放超過 80%」之具體判定為**累計覆蓋率**：學員實際播放過之區段聯集 ÷ 影片總長 ≥ 80%。允許學員拖動進度條跳轉播放，但**只有實際播放過的區段計入**（直接拉到結尾不產生觀看區段、不算看過）；允許倍速播放（播放器提供 0.75 / 1 / 1.25 / 1.5 / 2，**上限 2 倍**，由 `ET_PARAM.VIDEO_PLAYBACK_MAX_RATE` 控制），**倍速播放之區段照算覆蓋率**（2 倍速實際看完全片 = 覆蓋率 100%）；同一區段重複觀看不會多算覆蓋率（區段聯集去重）。

**理由**:
- 「連續播放時長」會被誤認為要從頭看到尾，UX 差且學員若中途暫停需重來
- 「必須線性播放」不符成人學習習慣（成人常跳到熟悉段落確認重點）
- 累計覆蓋率（聯集去重）兼顧「真的看過」與「彈性」；區段以**影片時間軸**記錄，天然滿足「拉到底不算、倍速照算」

**邊界**:
- 學員反覆 seek 同一段不會多算覆蓋率
- 學員瀏覽器當機 / 斷網時，已 INSERT 之區段紀錄不會丟失（normalize 在下次離開時補做）

**排除方案**:
- 連續播放時長：學員中途暫停需重來，UX 差
- 必須線性播放：禁止 seek，不符使用者期望
- 播放時長累加（含重複）：學員可在同段反覆觀看湊夠時長，失去評估意義
- **「看完時間 − 開始時間」牆鐘時間計算**（2026-07-02 客戶原始提案）：頭尾相減無法分辨中間是播放或暫停——學員開影片後掛機暫停再拉到底即可通過（反而可作弊）；且 2 倍速認真看完僅花 50% 牆鐘時間會被誤判不通過（與「可 2 倍速觀看」自相矛盾）。經與客戶確認改採累計覆蓋率，其「拉到底不算看過」之意圖由覆蓋率天然達成

### 15. 樂觀鎖（同 owner 多裝置編輯）

**決策**: 同一教師於多裝置同時編輯同一課程時採**樂觀鎖**：每實體（課程 / 章節 / 教材 / 測驗 / 題目）DB 維護 `VERSION` 欄位，寫入時檢核版本，不等則拒絕並提示「內容已被其他裝置變更」。檢核粒度為**單一實體**，同課程不同章節之同時編輯不互相衝突。

**理由**:
- ET 編輯為低頻動作（教師偶爾編輯課程），衝突機率低
- 悲觀鎖（A 編輯時 B 僅可閱覽）UX 差，可能因 session timeout 卡住
- 樂觀鎖實作簡單，DB 加 VERSION 欄位 + 寫入時 WHERE VERSION = ?

**排除方案**:
- 不處理（後存覆蓋前存）：教師可能丟失變更
- 悲觀鎖：UX 差、session timeout 卡住風險
- 整課程鎖：影響不同章節之同時編輯，過嚴

### 16. 強制顯示正確答案（無教師設定關閉之選項）

**決策**: 學員提交測驗並完成自動閱卷後，於答題明細頁**強制顯示正確答案**（題幹原文 + 所有選項含正確答案標示 + 學員當次選擇 + 該題得分）。無教師設定可關閉之選項。學員「未及格但重考次數未用完」時仍顯示。

**理由**:
- 「學習導向」為設計優先；訓練系統不同於考試系統，目的是讓學員學會
- 教師設定關閉之選項會造成題庫一致性問題（不同測驗不同設定，學員混淆）
- 重考時依「測驗作答機制」重新洗牌題目順序與選項順序，不會因看過答案即可立刻答對（學員仍需理解才能於洗牌後正確選擇）

**排除方案**:
- 不顯示正確答案：學員不知對錯，學習成效差
- 教師可設定：題庫一致性問題、教師設定遺漏風險
- 僅及格時顯示：學員未及格更需要學習，反而看不到答案

### 17. 受訓單位標籤 ET 自持、不與 DM 共用（2026-07-02 新增）

**決策**: 原「關聯模組 / 使用者×業務模組對應」（ET_MODULE / ET_USER_MODULE）**移除**，改為**受訓單位標籤**：`ET_TAG`（標籤庫，管理者維護、可停用啟用、含特殊標籤「全體」）+ `ET_USER_TAG`（人×標籤）+ `ET_COURSE_TAG`（課程×標籤，多對多）。結構**參照 DM「可見對象/單位」**（DM_TAG AUDIENCE 組 + DM_USER_TAG）之成熟模式，但 **ET 自持一套、不與 DM 共用**資料。

**理由**:
- 客戶 2026-07-02 要求：課程以受訓單位（護理師 / 行政人員 / 軍人 / 醫檢師 / 全體）為邀請單位，一課程可多標籤
- ET 與 DM 使用者群**相交而不相同**；後續管理者可能於單一系統新增另一系統沒有的單位——共用會互相綁死（客戶確認採自持）
- 參照 DM 已驗證之標籤模型（IS_ALL 全體、停用不影響既有掛載），降低設計風險

**排除方案**:
- 共用 DM_TAG / DM_USER_TAG：「誰是護理師」單一來源雖佳，但兩系統使用者群不同、未來標籤清單將分歧；且 ET 需依賴 DM 服務
- 保留業務模組並映射至標籤：兩套並存混淆，模組概念於 ET 已無其他用途

### 18. 課程起訖時間 + 可逆狀態機（關閉後可再開課）（2026-07-02 新增）

**決策**: `ET_COURSE` 新增 `OPEN_START_AT` / `OPEN_END_AT`（閱課期間，發布時必填）。起始前學員不可見；到期系統自動關閉（SCHET002 每日檢查＋應用層存取時即時判定雙保險）；關閉後學員**唯讀回看**已學內容（不能再累積進度 / 作答 / 解鎖 / 填問卷）；教師可**再開課**（重設一組新起訖時間，狀態回 PUBLISHED，進度接續），關閉 / 再開可重複多次。原「停課不可逆終態」廢除。

**理由**:
- 客戶 2026-07-02 要求：課程需有起訖時間、到期關閉閱課、停課後仍可能再開課
- 「唯讀回看」兼顧「關閉閱課」與學員複習需求（客戶確認：不能再學 / 作答，但已完成內容可回看）
- 到期判定採「排程＋應用層即時判定」雙保險，避免排程間隔造成邊界誤差

**排除方案**:
- 關閉後完全不可見：與客戶確認之唯讀回看不符
- 僅手動關閉（無到期自動）：無法滿足「時間到自動關閉」
- 再開課沿用原起訖：原訖止已過即再度立刻關閉，邏輯矛盾，故強制重設

### 19. 發布時標籤自動邀請＋統一範本寄信（2026-07-02 新增）

**決策**: 課程發布時，系統取 `ET_COURSE_TAG` × `ET_USER_TAG` 對應人員（限具「學員」角色者；「全體」展開為全部學員角色者）之**聯集、去重**，批次寫入 ET_ENROLLMENT（來源 TAG_DEFAULT）並**每人寄一封**通知信；管理者**事後貼標**時自動補加入該標籤所有「已發布且未關閉」課程並寄**彙整一封**（列出所有新加入課程）。信件一律採管理者維護之**統一範本**（`ET_NOTIFY_TEMPLATE`），教師不可逐課編輯；收件對象一律為**學員個人 Email**（不設單位信箱——「通知各單位」已由「標籤＝單位、通知該標籤所有人」吸收）。原教師手動 Email 邀請**保留為補件手段**（臨時追加不在標籤內的人），同採統一範本。

**理由**:
- 客戶 2026-07-02 要求：發布課程自動邀請標籤對應人員並發信通知；原「模組預設帶入」為靜默加入不寄信，不符「新增課程→通知被調訓人員」
- 事後貼標彙整一封避免「一人被貼標籤即收到 N 門舊課程 N 封信」之信件爆量（客戶確認）
- 統一範本確保寄出內容一致（客戶確認：管理員改範本即可，教師不需逐課編輯）

**排除方案**:
- 通知各單位信箱（主管信箱 / 群組信箱）：需另建「單位↔信箱」對應；經確認標籤制已吸收「單位」概念，不需要
- 事後貼標不追溯：新進人員無法自動補上既有訓練課程，不符調訓意圖
- 教師可編輯信件內文：與統一範本決策矛盾（原 US8 設計移除）

### 20. 每週統計快照 + 分層提醒（0% 週提醒、截止前 3 天加急）（2026-07-02 新增）

**決策**: 新增兩支排程：`SCHET001`（每週一 10:00，`ET_PARAM` 可調）統計各開放中課程並寫入快照 `ET_WEEKLY_STAT`（課程×週次），寄週報（教師收自己課程、管理者收全域；內文摘要＋CSV 逐學員明細；含與上週比較）；同批對**進度 0%（完全未開始）**學員寄未看提醒（一人一信彙整）。`SCHET002`（每日）檢查課程到期自動關閉，並於**訖止前 3 天**（`ET_PARAM` 可調）對該課程**所有未完課**學員寄加急提醒（每課只寄一次）。已關閉 / 未到起始課程不納入；已完課學員不收提醒。

**理由**:
- 客戶 2026-07-02 要求：每週自動統計看課百分比並回傳、同時通知未看者
- 週提醒僅寄 0% 者：長期課程（如三個月）若「未完課就提醒」會週週轟炸所有人，提醒疲勞反被忽略（客戶確認選此方案，接受「開始後停滯者要到截止前 3 天才被叫醒」之盲點，加急信為最後防線）
- 週報呈現「平均進度%＋人數分布＋與上週比較」而非只有完課率：長期課程前期完課率必為 0，無參考價值（客戶確認）
- 快照留存（append-only）支撐「與上週比較」與歷史回查
- 加急提醒每日排程獨立於週排程：各課程訖止日不同，週排程會漏

**排除方案**:
- 每週提醒所有未完課者：信量爆炸、提醒疲勞
- 週提醒門檻設 80%：卡在 80–99% 者永不再被提醒，但完課要求 100%，邏輯矛盾
- 停滯 N 週再提醒（0% ＋ 停滯者）：較完整但客戶選擇最簡方案（a）
- 不留快照即時算：無法呈現「與上週比較」、歷次統計不可回查

### 21. 課後回饋問卷（複用測驗結構模式、具名、一人一次、填答後凍結）（2026-07-02 新增）

**決策**: 新增課後回饋問卷：`ET_SURVEY`（一課程 0～1 份）+ `ET_SURVEY_QUESTION`（一律單選）+ `ET_SURVEY_OPTION`（選項教師自訂）+ `ET_SURVEY_RESPONSE_M/_D`（**具名**填答，一人一次、送出後不可改）。學員**完課後**開放填寫、課程關閉後不可填；**不是完課條件**、不計入進度；問卷**已有任何填答後題目與選項凍結**（僅可停用）。教師於 US9「問卷結果」檢視各題選項分布統計＋逐學員具名明細。

**理由**:
- 客戶 2026-07-02 要求：課後滿意度問卷、每題單選（如滿意 / 普通 / 不滿意）、選項教師自編、具名、教師可看每位學員選項
- 表結構 / 編輯畫面 / 樂觀鎖完全比照測驗四表模式，開發成本最低
- 填答後凍結避免已填資料與題目對不上（比照 Attempt Snapshot 之資料一致性精神，但問卷無需逐份快照——凍結整份問卷更簡單）

**排除方案**:
- 匿名問卷：客戶明確要求可看「每位學員」的選項（具名）
- 問卷計入完課條件：客戶確認不算；且會使完課定義複雜化
- 允許修改已填答問卷：統計結果不穩定、稽核困難

### 22. 每次作答明細開放教師與學員檢視（2026-07-02 新增）

**決策**: 測驗**每一次** attempt 之逐題明細（題目、學員作答、正確答案、對錯、該題得分）開放：**學員**可回看自己所有 attempt；**教師**於 US9 點擊學員展開歷次 attempt 清單並點入單次明細。明細依各 attempt 之快照（`ET_QUIZ_ATTEMPT_D` snapshot 欄位）渲染，與作答當下一致。

**理由**:
- 客戶 2026-07-02 要求：教師能看到每位學員的考試作答明細（不只平均成績）；並確認為「每次」而非僅最高分那次
- 資料本已完整保存於 ET_QUIZ_ATTEMPT_D（快照），僅需補查詢與 UI，無資料模型變更
- 依快照渲染避免題目事後被修改造成明細失真

**排除方案**:
- 僅開放最高分 attempt：客戶原傾向此案，後改為每次皆可看
- 教師可看、學員僅看最近一次：學員複習需求同樣存在，且資料已在

## 與其他模組之關係決策

| 議題 | 決策 |
|------|------|
| ET 與 DM 帳號共用範圍 | 共用 USER 主檔 + 密碼變更同步；角色與受訓單位標籤各自管理（標籤不共用，per #17）|
| ET 與 DP 之依賴 | 完全不依賴；ET_PARAM 自管，不呼叫 DP Service |
| ET 與主系統業務模組（BC / CP / TL / BS / MA / LB） | 完全不依賴；受訓單位標籤（ET_TAG）為 ET 自定，不對應主系統任何 table |
| ET 與 Email Server | 透過 SMTP 外部介接；不引入 MQ 或事件匯流排；排程（SCHET001 / SCHET002）由部署環境 cron 觸發 |
