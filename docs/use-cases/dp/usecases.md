# 平台模組（DP）使用案例（UCDP）

> 本文件依據 [../../requirements/RQDP.md](../../requirements/RQDP.md) 整理。
> **RQDP.md 全篇為分析資料、無 RFP RQ 編號**（TBMS 的 RQDP 屬主專案「資訊模組」，不沿用），故本文件「關聯需求」欄以 **RQDP 功能章節**對應，並於末段附「UC → RQDP 章節對照」取代 RQ 追蹤矩陣。
> 平台模組（DP）為 EDMS 共用層 + **統一管理後台**，ET / DM 共用其帳號、認證、稽核，以及系統參數 / 清單定義 / 通知範本 / 排程之**儲存與引擎**；所有管理畫面（參數與清單、通知範本、角色指派、帳號、稽核）集中於 DP 後台、按模組過濾。**「有哪些」的定義（分類 / 標籤名稱 / func_name / 參數值）存 `DP_PARAM`、「誰配誰」的關聯 / 指派與角色判定留各模組**；平台不做全域 RBAC（角色能力與判定在模組 enforce）。DP 後台**無獨立管理員角色**，由 **ET 管理者 / DM 管理者**進入。（2026-07-08 集中化與統一管理後台決策）

## 整體作業流程（角色視角）

```
一般使用者
──────────
1. UCDP002 首次使用自助註冊（即用，自動取得 ET 學員預設角色；其他角色由管理者開通）
2. UCDP001 以帳號 / 密碼登入（核發 JWT）；忘記密碼走 UCDP003
3. UCDP004 維護自己的個人資料（姓名 / 帳號 Email / 密碼）

ET / DM 管理者（DP 統一後台，按模組過濾；共用項兩者皆可）
──────────
1. UCDP005 使用者管理（建立 / 停用 / 啟用 / 基本資料 / 解鎖帳號）＊共用項
2. UCDP006 系統參數與清單維護（平台級 `DP_` 共用；`ET_`/`DM_` 參數與清單定義按模組過濾）
3. UCDP010 權限管理（於後台同時指派各模組角色與標籤/可見對象，寫入模組表）
4. UCDP011 通知範本維護（編輯 `DP_NOTIFY_TEMPLATE`，按 `MODULE` 過濾）
5. UCDP007 查詢操作記錄（稽核）＊共用項，兩者皆可查全部
6. UCDP008 排程總覽唯讀檢視（job / 上次執行結果）＊共用項

系統自動 / 發信引擎
────────
1. UCDP001 登入失敗達上限自動鎖定；逾時自動解鎖
2. UCDP008 依 `DP_SCHEDULE` 註冊之排程 job 執行（ET 週報 / 提醒、DM KPI 週報等）
3. UCDP009 各模組呼叫平台發信服務 → 依 `DP_NOTIFY_TEMPLATE` 範本渲染 → 經 outbox 寄送
```

## 共通規則

以下規則適用於所有使用案例，各 UC 不再重複：

- **帳號共用**：ET / DM 共用 `DP_USER`，使用者註冊 / 登入一次即可存取兩系統
- **認證方式**：簡單 JWT（驗證帳密後核發 access token），不採 MFA、不採 Refresh Token / 伺服器端 session；登出＝前端丟棄 token
- **統一管理後台**：平台為 EDMS 的**統一管理後台**，所有管理畫面（系統參數與清單、通知範本、權限管理（角色 + 標籤/可見對象指派）、帳號管理、操作記錄）集中於 DP，**無獨立管理員角色**，由 **ET 管理者 / DM 管理者**進入，畫面內容按模組過濾
- **權限分開**：平台**不做全域 RBAC**；ET / DM 之業務角色（教師 / 學員 / 編輯者…）之能力與判定（某角色能做什麼）皆由各模組 enforce（如 ET07）；角色**指派畫面**設於 DP 後台（按模組過濾），但指派**資料寫入各模組角色表**（`ET_USER_ROLE` / `DM_USER_ROLE`）
- **參數與清單定義集中 DP_PARAM、按模組過濾**：全平台系統參數**與清單定義**（分類 / func_name / 標籤名稱 / 可見對象值）統一存 `DP_PARAM`（`PARAM_ID` 前綴 `DP_`/`ET_`/`DM_`；需鎖定之碼用 `DETAIL_LOCK`），通知範本存 `DP_NOTIFY_TEMPLATE`（`MODULE` 分模組）；「定義（有哪些）」在 DP、「關聯 / 指派（誰配誰）」與角色判定留各模組。維護畫面統一於 DP 後台：模組項（前綴 `ET_`/`DM_`、`MODULE=ET`/`DM`）按模組過濾、互不可見，**共用項**（平台級無前綴 `DP_` 參數）ET / DM 管理者皆可存取；平台提供唯讀參數服務與唯一發信服務
- **稽核集中**：帳號建立、密碼重置、角色 / 資料異動、登入登出等統一寫入 `DP_AUDIT_LOG`（append-only，不可竄改刪除）；稽核為共用項，ET / DM 管理者皆可查全部
- **標準欄位**：EDMS 單一組織，無站點 / 院區維度；標準欄位為 CREATED_USER/DATE、UPDATED_USER/DATE、RES_ID、DELETED
- **排程集中**：平台提供排程引擎 + `DP_SCHEDULE` 註冊表 + `DP_SCHEDULE_LOG`；各模組 job 於此登錄，job 需要業務資料時反向 import 模組 service
- **資安**：密碼一律雜湊；身分驗證資訊加密傳輸；使用者 Email / 姓名為一般個資、不加密儲存

## 使用案例描述

### UCDP001 使用者登入 / 登出

- **主要角色**：全體使用者 / 系統自動（鎖定判定）
- **關聯需求**：登入認證、帳號鎖定（RQDP.md）
- **前置條件**：使用者已有啟用中帳號
- **主流程**：
  1. 使用者於登入頁輸入帳號（Email）+ 密碼
  2. 系統查 `DP_USER` 驗證帳密
  3. 驗證通過 → 核發 JWT access token（TTL 由參數控制）、重設失敗計數、寫登入稽核（LOGIN）
  4. 登入後由各模組依角色導向預設首頁（ET：管理者→ET07 / 教師→ET01 / 學員→ET04）
  5. 登出：前端丟棄 token，寫登出稽核（LOGOUT）
- **替代流程**：
  - 帳號不存在 → 提示「查無此帳號，請先註冊」
  - 密碼錯誤 → 失敗計數 +1；達上限（參數，預設 5）→ 自動鎖定帳號，提示已鎖定
  - 帳號已鎖定 / 已停用 / 閒置逾 90 日 → 拒絕登入並提示
  - JWT 逾時或閒置逾時 → 需重新登入
- **後置條件**：使用者取得有效 JWT；或帳號被鎖定待解鎖（UCDP005 / 逾時自動）

---

### UCDP002 使用者自助註冊

- **主要角色**：訪客（未登入者）
- **關聯需求**：使用者 / 帳號管理（RQDP.md）
- **前置條件**：無（登入頁可進入註冊）
- **主流程**：
  1. 訪客於登入頁點「註冊」，填寫帳號（Email）、姓名、密碼、確認密碼
  2. 系統檢核 Email 未被註冊、密碼兩次一致且符合複雜度
  3. 建立 `DP_USER` 紀錄（密碼雜湊），並自動授予 **ET 學員** 角色（唯一預設角色，透過 ET service 寫入 `ET_USER_ROLE`）；其他角色（ET 教師 / 管理者、DM 全部四角色含閱覽者）由管理者於 UCDP010 開通
  4. 提示註冊成功、跳回登入頁（帳號即可登入）
- **替代流程**：
  - Email 已被註冊 → 提示改走登入或忘記密碼
  - 密碼不符複雜度 / 兩次不一致 → 提示錯誤
- **後置條件**：新帳號建立且可登入

---

### UCDP003 忘記密碼

- **主要角色**：使用者本人 / eMail Server
- **關聯需求**：忘記密碼（RQDP.md）
- **前置條件**：使用者記得註冊之 Email
- **主流程**：
  1. 使用者於登入頁點「忘記密碼」，輸入帳號（Email）
  2. 系統產生具時效之一次性重設 token（TTL 參數，預設 30 分鐘），寄重設信
  3. 使用者點信中連結 → 系統驗證 token 有效性與期限
  4. 使用者輸入新密碼（檢核複雜度與重複性）→ 更新 `DP_USER`、記錄稽核
- **替代流程**：
  - 不論帳號是否存在皆回覆相同訊息（防帳號列舉）
  - token 逾期 / 無效 → 提示「連結已失效，請重新申請」
- **後置條件**：密碼更新，使用者可以新密碼登入

---

### UCDP004 個人資料維護

- **主要角色**：使用者本人 / eMail Server（Email 變更驗證）
- **關聯需求**：密碼與帳號安全、使用者 / 帳號管理（RQDP.md）
- **前置條件**：使用者已登入
- **主流程**：
  1. 使用者進入個人資料頁，可維護姓名、帳號（Email）、密碼
  2. **姓名**：直接編輯儲存，同步寫 `DP_USER`（ET / DM 兩端生效）
  3. **帳號（Email）變更**：寄驗證信至新 Email，點連結後切換（延遲生效）；期間舊 Email 仍可登入
  4. **密碼變更**：輸入舊密碼 + 新密碼 + 確認新密碼，驗證舊密碼正確、兩次一致、符合複雜度與重複性後更新
  5. 異動寫入稽核
- **替代流程**：
  - 舊密碼錯誤 → 提示「舊密碼不正確」
  - 新密碼兩次不一致 / 不符複雜度 → 提示錯誤
  - Email 驗證連結逾期 → 變更作廢，舊 Email 維持
- **後置條件**：個人資料更新並同步兩系統

---

### UCDP005 使用者管理

- **主要角色**：ET / DM 管理者（帳號管理為共用項，任一管理者皆可操作）
- **關聯需求**：使用者 / 帳號管理、帳號鎖定（RQDP.md）
- **前置條件**：操作者具 ET 或 DM 管理者身分並已登入
- **主流程**：
  1. 管理者於 DP 後台使用者管理頁查詢使用者清單
  2. 可執行：建立帳號、停用（Disable）/ 啟用（Enable）、維護基本資料、手動解鎖被鎖定帳號（角色指派不在此，見 UCDP010）
  3. 帳號權限以最小權限為原則；帳號唯一識別、不得共用
  4. 所有異動寫入稽核
- **替代流程**：
  - 停用帳號 → ET / DM 兩端同步失效
  - 連續 90 日未登入之帳號由系統自動鎖定 / 禁用（管理者可解鎖）
- **後置條件**：帳號狀態更新；稽核留痕

---

### UCDP006 系統參數與清單維護

- **主要角色**：ET / DM 管理者（平台級 `DP_` 為共用項、兩者皆可改；模組級 `ET_`/`DM_` 按模組過濾）
- **關聯需求**：系統參數與清單定義（RQDP.md）
- **前置條件**：操作者具 ET 或 DM 管理者身分並已登入
- **主流程**：
  1. 管理者於 DP 後台系統參數 / 清單頁選取參數或清單項；畫面按模組過濾——ET 管理者見 `ET_` 前綴、DM 管理者見 `DM_` 前綴，平台級無前綴 `DP_` 兩者皆見可改，模組項互不可見
  2. 維護**系統參數**（如平台級密碼策略、登入失敗鎖定次數 / 時間、密碼重設 token TTL、JWT TTL、閒置逾時、發信引擎調校；模組級影片大小、催辦門檻、排程時間等）與**清單定義**（文件分類、func_name、標籤名稱、可見對象值、DM 檢索標籤〔適用模組 / 文件性質 / 法規關聯〕等；清單型以一個 `PARAM_ID` 下多筆 `PARAM_KEY` 表達）
  3. 系統驗證合法性後儲存（寫 `DP_PARAM`；需鎖定之碼如文件分類碼建立後不可改，以 `DETAIL_LOCK` 標記），記錄異動歷程
- **替代流程**：值不合法 → 提示並拒絕儲存
- **後置條件**：參數 / 清單定義更新並即時生效
- **備註**：本 UC 只管「定義（有哪些）」；套用清單之**業務規則**（分類碼嵌入 `DOC_ID`、`func_name` 唯一手冊檢核、標籤「全體」`IS_ALL` 展開與自動邀請）與「誰配誰」的關聯 / 指派（文件屬哪分類、課程 × 標籤、使用者 × 標籤）由各模組於讀取後執行、留各模組業務表；平台另提供唯讀查詢服務供各模組讀取

---

### UCDP007 操作記錄查詢（稽核）

- **主要角色**：ET / DM 管理者（稽核為共用項，兩者皆可查全部）
- **關聯需求**：操作記錄（稽核）（RQDP.md）
- **前置條件**：操作者具 ET 或 DM 管理者身分並已登入
- **主流程**：
  1. 管理者以多條件（操作者、期間**起訖**、模組、操作類別等）查詢 `DP_AUDIT_LOG`（含登入等不分模組事件，兩管理者皆可查全部）
  2. 檢視操作歷程與異動前後值（JSONB）
  3. 匯出 CSV
- **替代流程**：
  - 稽核日誌 append-only，任何人不得於介面竄改 / 刪除
- **後置條件**：取得稽核查詢結果 / 匯出檔
- **備註**：ET / DM 之**資安稽核事件**（帳號 / 角色權限 / 系統操作）亦寫入同一 `DP_AUDIT_LOG` 於此統一查詢；**業務歷程**（DM 文件變更歷程、閱讀、ET 學習/作答紀錄）留各模組、面向使用者查詢，不在此

### UCDP008 排程作業執行與總覽

- **主要角色**：系統自動 / ET / DM 管理者（總覽唯讀檢視，共用項）
- **關聯需求**：排程基礎建設（RQDP.md）
- **前置條件**：各模組排程 job 已登錄於 `DP_SCHEDULE`（cron + 執行程式參照 + 啟停），job handler 已向引擎註冊
- **主流程**：
  1. 平台排程引擎依 `DP_SCHEDULE` 之 cron 觸發到期之 job（多實例部署時由 leader 實例執行）
  2. 呼叫該 job；job 需要業務資料時反向 import 所屬模組 service（如 ET 週報 / 提醒、DM KPI 週報）
  3. 以 `DP_SCHEDULE_LOG` 記錄排程執行歷程（起訖時間、成功 / 失敗、錯誤訊息）
- **替代流程**：
  - job 執行失敗 → 記錄錯誤；不影響其他 job；補跑機制由各模組視需要處理
- **後置條件**：排程作業完成並留執行紀錄
- **備註**：平台提供引擎 + `DP_SCHEDULE` 註冊表 + 執行紀錄；排程時間等參數存 `DP_PARAM`（前綴分模組），於 DP 後台由各模組管理者維護（見 UCDP006）。DP 後台另提供**排程總覽唯讀畫面**（共用項；列 job / cron / 啟停狀態 / 上次執行結果與歷程，**不可於 UI 啟停或手動補跑**，啟停由 DB / 部署管理，2026-07-08 釐清）；平台自身排程 `SCHDP001`（每日：閒置帳號禁用 + 密碼效期提醒）亦於此登錄

### UCDP009 通知發送服務

- **主要角色**：系統自動 / 發信引擎（外部 eMail Server）
- **關聯需求**：通知範本與發信引擎（RQDP.md）
- **前置條件**：對應 `TEMPLATE_CODE` 已存在於 `DP_NOTIFY_TEMPLATE` 且為啟用狀態
- **主流程**：
  1. 各模組於業務事件（課程邀請、文件送審、密碼重設…）呼叫平台發信服務 `send_email(recipients, template_code, params)`
  2. 平台以 `template_code` 取 `DP_NOTIFY_TEMPLATE` 範本、代入變數渲染主旨 / 內文
  3. 寫入 `DP_EMAIL_LOG` outbox（`CALLER_MODULE`、收件人、狀態 PENDING）
  4. 背景 worker 依速率 / 重試參數（平台級 `DP_`）透過外部 SMTP 寄送，更新狀態（SENT / FAILED），失敗達門檻告警
- **替代流程**：
  - 範本停用（IS_ENABLED=false）→ 該類信件不寄（觸發事件照常運作）
  - 寄送失敗 → 依重試上限重試；逾上限標記 FAILED 並計入失敗告警比例
- **後置條件**：信件寄出並於 `DP_EMAIL_LOG` 留結果
- **備註**：本 UC 為發信**服務**（無畫面）；範本之**編輯**由 UCDP011 於 DP 後台處理（按 `MODULE` 過濾）；DP 自身系統信（`MODULE=DP`）主旨 / 內文可編輯、但不可停用 / 刪除

---

### UCDP010 權限管理（角色 + 標籤 / 可見對象指派）

- **主要角色**：ET / DM 管理者（各自指派自己模組之角色與標籤，按模組過濾、互不可見）
- **關聯需求**：權限管理（角色 + 標籤 / 可見對象指派）（RQDP.md）
- **前置條件**：操作者具 ET 或 DM 管理者身分並已登入
- **主流程**：
  1. 管理者於 DP 後台「權限管理」畫面查詢使用者，於**同一列同時**指派 / 調整角色與標籤（對齊 ET / DM 現行權限管理），畫面按模組過濾：
     - **ET**：角色（管理者 / 教師 / 學員）＋**受訓單位標籤**（供課程發布自動邀請）
     - **DM**：角色（管理者 / 編輯者 / 審核者 / 閱覽者）＋**可見對象/單位授權**（主要對閱覽者有意義）
     - 角色與標籤為獨立維度，可各自獨立設定
  2. 指派結果透過模組 service 寫入模組表：角色 → `ET_USER_ROLE` / `DM_USER_ROLE`；標籤指派 → `ET_USER_TAG`；可見對象授權 → DM 授權表。平台不做全域 RBAC、不定義角色能力
  3. 角色 / 標籤指派異動屬資安事件，寫入 `DP_AUDIT_LOG`
- **替代流程**：角色種類為固定 enum（無「新增角色種類」）；同一使用者可多角色，權限取聯集
- **後置條件**：使用者角色與標籤更新（寫入模組表）；能力判定 / enforce 由各模組於業務操作時執行
- **備註**：指派**畫面**在 DP 後台，指派**資料與判定**留各模組；標籤/可見對象之**定義清單**在 `DP_PARAM`（見 UCDP006），此處只做「指派」（誰配誰）

---

### UCDP011 通知範本維護

- **主要角色**：ET / DM 管理者（各自編輯自己模組範本，按 `MODULE` 過濾；DP 系統信為共用項）
- **關聯需求**：通知範本與發信引擎（RQDP.md）
- **前置條件**：操作者具 ET 或 DM 管理者身分並已登入
- **主流程**：
  1. 管理者於 DP 後台「通知範本」畫面選取範本；畫面按 `MODULE` 過濾——ET 管理者見 `MODULE=ET`、DM 管理者見 `MODULE=DM`；DP 系統信（`MODULE=DP`）兩者皆可見
  2. 編輯主旨、內文、可用變數、管道（Email / 站內 / 兩者）、啟用停用；儲存寫入 `DP_NOTIFY_TEMPLATE`（版本樂觀鎖）
  3. 異動寫入操作記錄
- **替代流程**：
  - DP 系統信（`MODULE=DP`，密碼重設 / 帳號變更驗證）：兩管理者皆可編輯主旨 / 內文，但**不可停用 / 刪除**（帳號安全信）
  - 版本衝突（他人已改）→ 提示重載後再存
- **後置條件**：範本更新；後續 UCDP009 發信服務即以最新範本渲染
- **備註**：範本編輯集中於 DP 後台，取代原「各模組系統設定頁編輯」；發信服務（渲染 + outbox）為 UCDP009
