# 平台模組（DP）使用案例（UCDP）

> 本文件依據 [../../requirements/RQDP.md](../../requirements/RQDP.md) 整理。
> **RQDP.md 全篇為分析資料、無 RFP RQ 編號**（TBMS 的 RQDP 屬主專案「資訊模組」，不沿用），故本文件「關聯需求」欄以 **RQDP 功能章節**對應，並於末段附「UC → RQDP 章節對照」取代 RQ 追蹤矩陣。
> 平台模組（DP）為 EDMS 共用層，ET / DM 共用其帳號、認證、稽核、**系統參數 / 通知範本 / 排程之儲存與引擎**；各模組業務角色與權限、以及參數值 / 範本文案 / job 邏輯等業務內容**仍各自維護**（權限分開做；平台不碰業務角色；集中的只是儲存與引擎，維護 UI 按模組隔離）。（2026-07-08 集中化決策）

## 整體作業流程（角色視角）

```
一般使用者
──────────
1. UCDP002 首次使用自助註冊（即用，自動取得所屬模組預設角色）
2. UCDP001 以帳號 / 密碼登入（核發 JWT）；忘記密碼走 UCDP003
3. UCDP004 維護自己的個人資料（姓名 / 帳號 Email / 密碼）

平台管理員（IT）
──────────
1. UCDP005 使用者管理（建立 / 停用 / 啟用 / 基本資料 / 解鎖帳號）
2. UCDP006 維護平台級 `DP_` 系統參數（密碼策略、鎖定、發信引擎調校等）
3. UCDP007 查詢全平台操作記錄（稽核）

系統自動 / 發信引擎
────────
1. UCDP001 登入失敗達上限自動鎖定；逾時自動解鎖
2. UCDP008 依 `DP_SCHEDULE` 註冊之排程 job 執行（ET 週報 / 提醒、DM KPI 週報等）
3. UCDP009 各模組呼叫平台發信服務 → 依 `DP_NOTIFY_TEMPLATE` 範本渲染 → 經 outbox 寄送
```

## 共通規則

以下規則適用於所有使用案例，各 UC 不再重複：

- **帳號共用**：ET / DM 共用 `DP_USER`，使用者註冊 / 登入一次即可存取兩系統
- **認證方式**：簡單 JWT（驗證帳密後核發 access token），不採 MFA、不採 Refresh Token / 伺服器端 session；登出＝前端丟棄 token
- **權限分開**：平台只認「帳號」與「是否為平台管理員」（`DP_USER.IS_PLATFORM_ADMIN`，非業務角色體系）；ET / DM 之業務角色（教師 / 學員…）之定義、指派、判定皆由各模組自理（如 ET07），不在本模組；平台不做全域選單 RBAC
- **稽核集中**：帳號建立、密碼重置、角色 / 資料異動、登入登出等統一寫入 `DP_AUDIT_LOG`（append-only，不可竄改刪除）
- **參數 / 範本集中、按模組隔離**：全平台系統參數存 `DP_PARAM`（前綴 `DP_`/`ET_`/`DM_`）、通知範本存 `DP_NOTIFY_TEMPLATE`（`MODULE` 分模組）；平台提供唯讀參數服務與唯一發信服務；維護 UI 各模組自理，平台管理員只看平台級 `DP_`、看不到 ET / DM 業務參數與範本
- **標準欄位**：EDMS 單一組織，無站點 / 院區維度；標準欄位為 CREATED_USER/DATE、UPDATED_USER/DATE、RES_ID、DELETED
- **排程集中**：平台提供排程引擎 + `DP_SCHEDULE` 註冊表 + `DP_SCHEDULE_LOG`；各模組 job 於此登錄，job 需要業務資料時反向 import 模組 service
- **資安**：密碼一律雜湊；身分驗證資訊加密傳輸；使用者 Email / 姓名為一般個資、不加密儲存

## 使用案例描述

### UCDP001 使用者登入 / 登出

- **主要角色**：全體使用者 / 系統自動（鎖定判定）
- **關聯需求**：登入認證、帳號鎖定（RQDP.md）
- **前置條件**：使用者已有啟用中帳號
- **主流程**：
  1. 使用者於登入頁輸入帳號（Email）+ 密碼
  2. 系統查 `DP_USER` 驗證帳密
  3. 驗證通過 → 核發 JWT access token（TTL 由參數控制）、重設失敗計數、寫登入稽核（LOGIN）
  4. 登入後由各模組依角色導向預設首頁（ET：管理者→ET07 / 教師→ET01 / 學員→ET04）
  5. 登出：前端丟棄 token，寫登出稽核（LOGOUT）
- **替代流程**：
  - 帳號不存在 → 提示「查無此帳號，請先註冊」
  - 密碼錯誤 → 失敗計數 +1；達上限（參數，預設 5）→ 自動鎖定帳號，提示已鎖定
  - 帳號已鎖定 / 已停用 / 閒置逾 90 日 → 拒絕登入並提示
  - JWT 逾時或閒置逾時 → 需重新登入
- **後置條件**：使用者取得有效 JWT；或帳號被鎖定待解鎖（UCDP005 / 逾時自動）

---

### UCDP002 使用者自助註冊

- **主要角色**：訪客（未登入者）
- **關聯需求**：使用者 / 帳號管理（RQDP.md）
- **前置條件**：無（登入頁可進入註冊）
- **主流程**：
  1. 訪客於登入頁點「註冊」，填寫帳號（Email）、姓名、密碼、確認密碼
  2. 系統檢核 Email 未被註冊、密碼兩次一致且符合複雜度
  3. 建立 `DP_USER` 紀錄（密碼雜湊），並自動取得所屬模組之預設角色（如 ET 學員）
  4. 提示註冊成功、跳回登入頁（帳號即可登入）
- **替代流程**：
  - Email 已被註冊 → 提示改走登入或忘記密碼
  - 密碼不符複雜度 / 兩次不一致 → 提示錯誤
- **後置條件**：新帳號建立且可登入

---

### UCDP003 忘記密碼

- **主要角色**：使用者本人 / eMail Server
- **關聯需求**：忘記密碼（RQDP.md）
- **前置條件**：使用者記得註冊之 Email
- **主流程**：
  1. 使用者於登入頁點「忘記密碼」，輸入帳號（Email）
  2. 系統產生具時效之一次性重設 token（TTL 參數，預設 30 分鐘），寄重設信
  3. 使用者點信中連結 → 系統驗證 token 有效性與期限
  4. 使用者輸入新密碼（檢核複雜度與重複性）→ 更新 `DP_USER`、記錄稽核
- **替代流程**：
  - 不論帳號是否存在皆回覆相同訊息（防帳號列舉）
  - token 逾期 / 無效 → 提示「連結已失效，請重新申請」
- **後置條件**：密碼更新，使用者可以新密碼登入

---

### UCDP004 個人資料維護

- **主要角色**：使用者本人 / eMail Server（Email 變更驗證）
- **關聯需求**：密碼與帳號安全、使用者 / 帳號管理（RQDP.md）
- **前置條件**：使用者已登入
- **主流程**：
  1. 使用者進入個人資料頁，可維護姓名、帳號（Email）、密碼
  2. **姓名**：直接編輯儲存，同步寫 `DP_USER`（ET / DM 兩端生效）
  3. **帳號（Email）變更**：寄驗證信至新 Email，點連結後切換（延遲生效）；期間舊 Email 仍可登入
  4. **密碼變更**：輸入舊密碼 + 新密碼 + 確認新密碼，驗證舊密碼正確、兩次一致、符合複雜度與重複性後更新
  5. 異動寫入稽核
- **替代流程**：
  - 舊密碼錯誤 → 提示「舊密碼不正確」
  - 新密碼兩次不一致 / 不符複雜度 → 提示錯誤
  - Email 驗證連結逾期 → 變更作廢，舊 Email 維持
- **後置條件**：個人資料更新並同步兩系統

---

### UCDP005 使用者管理

- **主要角色**：平台管理員
- **關聯需求**：使用者 / 帳號管理、帳號鎖定（RQDP.md）
- **前置條件**：操作者具平台管理員身分（`IS_PLATFORM_ADMIN`）並已登入
- **主流程**：
  1. 平台管理員於使用者管理頁查詢使用者清單
  2. 可執行：建立帳號、停用（Disable）/ 啟用（Enable）、維護基本資料、手動解鎖被鎖定帳號（角色指派不在此，由各模組自理）
  3. 帳號權限以最小權限為原則；帳號唯一識別、不得共用
  4. 所有異動寫入稽核
- **替代流程**：
  - 停用帳號 → ET / DM 兩端同步失效
  - 連續 90 日未登入之帳號由系統自動鎖定 / 禁用（管理者可解鎖）
- **後置條件**：帳號狀態更新；稽核留痕

---

### UCDP006 系統參數維護（平台級）

- **主要角色**：平台管理員
- **關聯需求**：系統參數（RQDP.md）
- **前置條件**：操作者具平台管理員身分（`IS_PLATFORM_ADMIN`）並已登入
- **主流程**：
  1. 平台管理員於 DP 系統參數頁檢視平台級 `DP_` 參數（頁面**只顯示** `DP_` 前綴，不含 ET / DM 模組參數）
  2. 編輯值（如密碼策略、登入失敗鎖定次數 / 時間、密碼重設 token TTL、JWT TTL、閒置逾時、發信引擎調校）
  3. 系統驗證合法性後儲存（寫 `DP_PARAM`），記錄異動歷程
- **替代流程**：值不合法 → 提示並拒絕儲存
- **後置條件**：平台級參數更新並即時生效
- **備註**：ET / DM 模組參數（影片大小、催辦門檻、排程時間等）雖同存 `DP_PARAM`（前綴 `ET_`/`DM_`），但由各模組管理者於自己的系統設定頁維護、DP 頁不顯示；平台提供唯讀查詢服務供各模組讀取

---

### UCDP007 操作記錄查詢（稽核）

- **主要角色**：平台管理員（或特定授權人員）
- **關聯需求**：操作記錄（稽核）（RQDP.md）
- **前置條件**：操作者具查詢稽核之授權並已登入
- **主流程**：
  1. 管理者以多條件（操作者、期間**起訖**、模組、操作類別等）查詢 `DP_AUDIT_LOG`
  2. 檢視操作歷程與異動前後值（JSONB）
  3. 匯出 CSV
- **替代流程**：
  - 稽核日誌 append-only，任何人不得於介面竄改 / 刪除
- **後置條件**：取得稽核查詢結果 / 匯出檔
- **備註**：ET / DM 之**資安稽核事件**（帳號 / 角色權限 / 系統操作）亦寫入同一 `DP_AUDIT_LOG` 於此統一查詢；**業務歷程**（DM 文件變更歷程、閱讀、ET 學習/作答紀錄）留各模組、面向使用者查詢，不在此

---

### UCDP008 排程作業執行

- **主要角色**：系統自動
- **關聯需求**：排程基礎建設（RQDP.md）
- **前置條件**：各模組排程 job 已登錄於 `DP_SCHEDULE`（cron + 執行程式參照 + 啟停），job handler 已向引擎註冊
- **主流程**：
  1. 平台排程引擎依 `DP_SCHEDULE` 之 cron 觸發到期之 job（多實例部署時由 leader 實例執行）
  2. 呼叫該 job；job 需要業務資料時反向 import 所屬模組 service（如 ET 週報 / 提醒、DM KPI 週報）
  3. 以 `DP_SCHEDULE_LOG` 記錄排程執行歷程（起訖時間、成功 / 失敗、錯誤訊息）
- **替代流程**：
  - job 執行失敗 → 記錄錯誤；不影響其他 job；補跑機制由各模組視需要處理
- **後置條件**：排程作業完成並留執行紀錄
- **備註**：平台提供引擎 + `DP_SCHEDULE` 註冊表 + 執行紀錄；排程時間等參數存 `DP_PARAM`（前綴分模組），由各模組維護

---

### UCDP009 通知發送服務

- **主要角色**：系統自動 / 發信引擎（外部 eMail Server）
- **關聯需求**：通知範本與發信引擎（RQDP.md）
- **前置條件**：對應 `TEMPLATE_CODE` 已存在於 `DP_NOTIFY_TEMPLATE` 且為啟用狀態
- **主流程**：
  1. 各模組於業務事件（課程邀請、文件送審、密碼重設…）呼叫平台發信服務 `send_email(recipients, template_code, params)`
  2. 平台以 `template_code` 取 `DP_NOTIFY_TEMPLATE` 範本、代入變數渲染主旨 / 內文
  3. 寫入 `DP_EMAIL_LOG` outbox（`CALLER_MODULE`、收件人、狀態 PENDING）
  4. 背景 worker 依速率 / 重試參數（平台級 `DP_`）透過外部 SMTP 寄送，更新狀態（SENT / FAILED），失敗達門檻告警
- **替代流程**：
  - 範本停用（IS_ENABLED=false）→ 該類信件不寄（觸發事件照常運作）
  - 寄送失敗 → 依重試上限重試；逾上限標記 FAILED 並計入失敗告警比例
- **後置條件**：信件寄出並於 `DP_EMAIL_LOG` 留結果
- **備註**：範本**編輯** UI 在各模組系統設定頁（ET09 / DM09，過濾自己 `MODULE`）；DP 自身固定系統信（`MODULE=DP`）不可編輯 / 停用
