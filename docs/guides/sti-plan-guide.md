# `/sti-plan` 使用指南

## 使用時機

當你**收到一個要開發的 Issue**，準備開工**之前**使用。指令會幫你做「實作前盤點」：把規格讀通、判斷規模、寫成規劃留言貼到 issue，並整理出真正需要問 SA 的問題。

> **為什麼要做**：開工前先把規格對齊、把不清楚的地方問清楚，比寫到一半才發現規格沒寫、回頭等 SA 省時得多。

## 三支命令怎麼搭

```
/sti-plan #432        ← 開工前盤點（寫規劃 + 提 SA Q 到 issue）
       │
       ▼  ⏳ 等 SA 回 SA Q（若有卡關的問題）
       │
/sti-implement #432   ← 開工（開分支 → TDD → PR）
```

| 命令 | 誰用 | 做什麼 | 動分支/程式碼？ |
|------|------|--------|----------------|
| `/sti-sa-precheck` | SA | 規格交付前自檢 | 否（只讀規格） |
| **`/sti-plan`** | **SD** | **開工前盤點，產出 issue 留言** | **否（只寫 issue 留言）** |
| `/sti-implement` | SD | 依規劃實作 | 是 |

> `/sti-plan` 不會開分支、不會改任何程式碼，它只在 issue 上留下「規劃留言」和「SA Q 留言」。

## 使用流程示範

**在 Claude Code 輸入：**

```
/sti-plan #432
```

---

**步驟 0：讀規格（自動）**

指令會自動讀：issue 本身、同模組已 close 的前例 issue（當模板）、spec / spec_us / data-model / contracts / wireframe，並調查跨模組依賴。過程不需操作。

```
已讀取：
  Issue #432（BS US3 出庫）
  前例：#420（當 18 節模板參考）
  docs/specs/bs/spec_us3.md、data-model.md、contracts/SRVBS002.md
  docs/wireframes/bs/index.html（screen-bs-03 區段）
✅ 規格讀取完成。
```

---

**步驟 1：判複雜度、選模板**

```
複雜度判定：FR 7 條、跨模組依賴有、獨有業務主題 2 個
→ 達複雜門檻，採 18 節規劃模板。
```

---

**步驟 2~4：寫規劃 + 幻覺檢查（自動）**

指令會寫好規劃留言，並把開發疑問逐題對照 spec 過濾掉「spec 已答 / SD 自決 / 可沿用前例」的部分，只留真正要問 SA 的。

---

**步驟 5：給你 review（重點：不會自己 post）**

```
盤點完成：
  規劃留言：18 節，/tmp/issue432/planning.md
  SA Q：初版 7 條 → 撤回 5 條 → 真 Q 2 條
    撤回摘要：Q3（spec_us3 已答）、Q5（屬 SD 自決：欄位形式）...
  跨模組依賴：SRVBS002 已交付 ✅

是否 post 到 issue #432？
  1. 確認 post（先規劃留言、後 SA Q 留言）
  2. 先修改規劃 / SA Q
  3. 取消
```

> 輸入 `1` 後，指令依序貼上兩則留言（規劃在前、SA Q 在後，讓 SA 回覆能接在問題下方）。

---

**步驟 6：建議下一步**

```
✅ 盤點完成。真 SA Q 不卡後端骨架。
   可直接開工：/sti-implement #432（Phase 1 後端可先做，Phase 2 等 SA 回 Q2）
```

---

## 看懂「下一步建議」

指令跑完會依 SA Q 是否擋住開發給你建議：

| 你看到 | 意思 |
|--------|------|
| `✅ 可直接開工：/sti-implement #N` | 沒卡關的問題，直接接著做 |
| `✅ 可先做 Phase 1 後端…Phase 2 等 SA` | 後端骨架不卡，前端細節等 SA |
| `⏳ 建議等 SA 回覆後再開工` | 問題卡到 schema/API 介面，先別動 |
| `⚠ 依賴 {工具/表} 未交付` | 卡跨模組依賴，先確認影響範圍 |

---

## 常見情況

| 情況 | 指令的處理方式 |
|------|---------------|
| issue 很簡單 | 步驟 1 自動選 10 節精簡模板 |
| 純 bug fix / 改樣式 / 跟版 | 屬「不適用盤點」，可直接用 `/sti-implement`，不需先 `/sti-plan` |
| 想先改規劃再貼 | 步驟 5 選 `2` |
| 不想貼上去 | 步驟 5 選 `3`，留言只留在本機 `/tmp` |

---

## 注意事項

- 指令**不會自己 post**——一定先給你 review，你按 `1` 確認才貼到 issue
- 規劃留言**先**於 SA Q 留言，順序固定，別手動顛倒
- 真 SA Q 會 **@ 這個 issue 的開立者**（通常是寫規格的 SA），不寫死特定帳號；若開立者不是 SA，指令會問你該 @ 誰。SA 回覆後若要改規格，由 SA 同步 `docs/specs`
- 命令檔放在 `.claude/commands/sti-plan.md`，跟著本專案版本控管
