執行以下步驟，為指定模組 / US 的規格做「交付前自檢」：在把 issue 交給 SD 開發前，逐項檢查 spec / data-model / contracts / wireframe 是否齊備且一致，並輸出缺口報告。

## 適用對象

此指令專為 SA（系統分析師）設計。目的：在規格交付 SD 開發**之前**自我檢查，讓 SD 開發時需要回頭問 SA 的問題趨近於零。

> 把這支 `.md` 放進你 spec repo 的 `.claude/commands/` 即可使用。

## 確認機制

需要使用者確認的環節一律提供**數字選項**：

| 使用者回覆 | 動作 |
|---------|------|
| 數字（如 `1`、`2`） | 執行對應選項 |
| 直接輸入文字 | 視為自訂內容（如自訂路徑） |
| `0` / `取消` | 停止並顯示「已取消。」 |

## 參數說明（$ARGUMENTS）

| 參數 | 說明 | 範例 |
|------|------|------|
| `模組代碼 US編號` | 指定模組與 US | `/sti-sa-precheck bs us4` |
| `模組代碼` | 只給模組，US 由互動選 | `/sti-sa-precheck bs` |
| 無參數 | 互動詢問模組與 US | `/sti-sa-precheck` |

---

## 執行步驟

### 步驟 1：定位規格檔

依 `$ARGUMENTS` 解析出 `{模組}` 與 `{US}`（缺的就詢問使用者）。

使用 Glob / Bash 探測規格目錄，預設結構為 `docs/specs/{模組}/`：

```bash
ls docs/specs/{模組}/ 2>/dev/null
ls docs/specs/{模組}/contracts/ 2>/dev/null
ls docs/wireframes/{模組}/ 2>/dev/null
```

對照出本次要檢查的檔案清單：

| 用途 | 預設路徑 |
|------|---------|
| 本 US 規格 | `docs/specs/{模組}/spec_us{N}.md` |
| 模組總覽 | `docs/specs/{模組}/spec.md` |
| 資料模型 | `docs/specs/{模組}/data-model.md` |
| 跨模組契約 | `docs/specs/{模組}/contracts/*.md` |
| Wireframe | `docs/wireframes/{模組}/index.html` |

若實際結構不同（找不到任一檔），顯示偵測到的目錄樹，詢問使用者：

```
偵測不到預設路徑的 {缺少的檔案}。請選擇：
  1. 此檔本 US 不適用（跳過該項檢查）
  2. 我提供正確路徑
```

確認後讀取所有存在的檔案，顯示：「✅ 已載入 N 個規格檔，開始自檢。」

---

### 步驟 2：A-1 spec 完整性檢查

讀 `spec_us{N}.md`（與 `spec.md` 相關段落），逐項判定：

| 檢查項 | 判定方式 |
|--------|---------|
| FR 編號清楚、無「待補 / TBD」 | grep `TBD`、`待補`、`待確認`、`???` |
| AC 可驗證、與 FR 對得上 | 逐條 AC 檢查是否量化（標出「適當 / 合理 / 盡量」等模糊措辭） |
| 訊息類型表完整 | 每訊息有：觸發情境、訊息文字、嚴重度（成功/警告/錯誤） |
| 異常 / 邊界情境有寫 | 是否只寫了 happy path |

---

### 步驟 3：A-2 data-model 檢查

讀 `data-model.md` 中本 issue 相關的 Table 段落：

| 檢查項 | 判定方式 |
|--------|---------|
| 欄位齊全 | 每欄位有：型別、可空、預設值、唯一性 |
| 業務規則寫明 | 狀態機轉移 / 軟刪除 cascade / 加密欄位 等是否描述 |

---

### 步驟 4：A-3 contracts 檢查（跨模組才需要）

若本 US **不涉及跨模組**，標記「N/A」並跳過。

否則讀 `contracts/*.md` 中本 issue 涉及的 SRV/API：

| 檢查項 | 判定方式 |
|--------|---------|
| 契約已定義 | req / resp 欄位、error 情境完整 |
| 關鍵點寫明 | 呼叫方向、權限、識別碼（如 site_id）來源 |
| 依賴狀態 | 是否依賴尚未交付的他模組工具（需在 issue 標註提醒 SD） |

---

### 步驟 5：A-4 wireframe 檢查

讀 `wireframes/{模組}/index.html`，grep 出本 US 對應的 screen-id 區段：

| 檢查項 | 判定方式 |
|--------|---------|
| 對應 screen 區段存在且可 grep | 找得到對應 screen-id |
| 狀態涵蓋 | 主要狀態 + 異常 / hover / 空資料 |
| 與訊息表一致 | 畫面提示文字對得上 spec 的訊息類型表 |

---

### 步驟 6：A-5 一致性交叉檢查（最容易漏）

跨檔對照，這是真正會擋住 SD 的地方：

| 交叉檢查 | 判定方式 |
|---------|---------|
| spec ↔ data-model | FR 提到的資料，欄位都建得起來 |
| spec ↔ wireframe | AC 要驗的東西，畫面都呈現得出來 |
| spec ↔ contracts | 跨模組行為，契約都覆蓋 |

逐項列出**對不上的地方**（例：「spec_us4 FR-3 提到『退回原因』，但 data-model 的 Table 無對應欄位」）。

---

### 步驟 7：輸出缺口報告

彙整成報告（繁體中文）：

```
# {模組} US{N} 規格交付自檢報告

## 逐項結果
| 項目 | 結果 | 缺口 |
|------|------|------|
| A-1 spec 完整性 | ✅ / ⚠ / ❌ | {具體缺什麼} |
| A-2 data-model | ✅ / ⚠ / ❌ | ... |
| A-3 contracts | ✅ / N/A / ❌ | ... |
| A-4 wireframe | ✅ / ⚠ / ❌ | ... |
| A-5 一致性交叉 | ✅ / ⚠ / ❌ | ... |

## 必補項（擋交付）
1. {具體缺口 + 建議怎麼補}

## 建議補強（不擋交付，但 SD 可能會問）
1. ...

## 結論
{二擇一}
✅ 規格齊備，可交付 SD 開發。
⚠ 有 N 項必補，建議先補規格再交付（比事後回 SD 問題省時）。
```

> **判定原則**：
> - `❌ 必補`：spec 未明寫且**影響業務行為**的缺口（SD 一定會回頭問）
> - `⚠ 建議`：屬 SD 自決範圍（UX / 實作選擇 / 欄位形式 / 檔案位置）但寫清楚更好
> - `✅`：齊備

---

## 共用判斷準則

### 複雜度分級（預判 issue 規模）

| 指標 | 簡單 | 複雜 |
|------|------|------|
| FR 數 | ≤ 5 | > 5 |
| AC 數 | ≤ 5 | > 5 |
| 訊息類型 | ≤ 5 | > 10 |
| 跨模組依賴 | 無 | 有 |
| 獨有業務邏輯主題 | 0 | ≥ 2 |

→ 越複雜，越要確保 A-3 / A-5 到位。

### 不適用自檢的情境

- 純 bug fix（根因明確、改動小）
- 純樣式調整（CSS / 文字）
- 跟版 / 套件升級
- 單純 refactor，不改外部行為
