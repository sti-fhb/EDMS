> **📌 Canonical Source**：本指令定義了 Issue 格式的唯一模板。其他指令（如 `/sti-implement`、`/sti-pr-review-fix`）若需建立 Issue，必須依照本模板的「Issue 內容模板」區塊格式。

執行以下步驟，以互動方式建立新的 GitHub Issue。

## 確認機制

所有需要使用者確認的環節，一律提供**數字選項**，使用者輸入數字即可：

| 使用者回覆 | 動作 |
|---------|------|
| 數字（如 `1`、`2`、`3`） | 執行對應選項 |
| 直接輸入文字 | 視為自訂內容 |
| `0` / `取消` | 停止並顯示「已取消，Issue 未建立。」 |

## Issue 內容模板

建立的 Issue body 須符合以下格式：

```
## 任務說明

{功能描述，說明背景與目的}

## 範圍

{列出需要實作的項目，前後端分開列出}

## 驗收條件

- [ ] {條件 1}
- [ ] {條件 2}

## 依賴

{若無依賴，填「無，可獨立開發」；若有，列出依賴的 Issue 編號與原因}

## 注意事項

- {特別限制或規範提醒}

## 相關文件

- {相關 SPEC 或規則檔路徑}
```

## 參數說明（$ARGUMENTS）

| 參數 | 說明 | 範例 |
|------|------|------|
| `標題文字` | Issue 標題的預設值（選填）| `/sti-issue-create 新增課程列表` |

---

## 執行步驟

### 步驟 1：解析參數

若 `$ARGUMENTS` 有值，將其作為 Issue 標題的預設值。

---

### 步驟 2：收集 Issue 資訊

逐步向使用者詢問以下資訊（每次一個問題，等待回答後再問下一個）：

| 欄位 | 說明 | 必填 |
|------|------|------|
| 標題 | Issue 標題，格式：`[模組代碼] 功能描述`，模組代碼對應下方「類型」（如 `[ET]`、`[DM]`）；基礎建設用 `[Foundation]`、跨模組用 `[跨模組]` | 必填 |
| 類型 |  ET 教育訓練模組 / DM 文件管理模組（介接） | 必填 |
| 任務說明 | 功能背景與目的 | 必填 |
| 範圍 | 需實作的項目（前端/後端） | 必填 |
| 驗收條件 | 完成標準，逐條列出 | 必填 |
| 依賴 | 依賴的 Issue 編號（無則填無） | 必填 |
| 注意事項 | 特別限制或規範（無則輸入「無」） | 選填 |
| 相關文件 | SPEC 或規則檔路徑（無則輸入「無」） | 選填 |
| 負責人 | GitHub 帳號，如 Sandra-168（無則輸入「無」） | 選填 |
| Label | 如 blocked（無則輸入「無」） | 選填 |

---

### 步驟 3：顯示預覽並確認

收集完所有資訊後，顯示完整預覽：
- 標題
- 完整 body 內容（依模板組合）
- 負責人、Label

詢問使用者：
```
  1. 確認建立
  2. 需要修改（請說明）
```

---

### 步驟 4：建立 Issue

使用者選擇 `1` 後，使用 Bash 工具執行：

```bash
gh issue create --title "{標題}" --body "{body}" [--assignee {帳號}] [--label {label}]
```

---

### 步驟 5：顯示結果

建立成功後顯示 Issue 編號與 URL。

```
✅ Issue 已建立：#{編號} {標題}
🔗 {URL}

下一步：執行 /sti-implement #{編號} 開始開發
```
